Amazon SES

Amazon SESのカスタムMAIL FROM設定、Route 53へのMX/TXTレコード登録、Linuxサーバー上のPostfixからSES SMTPへメールを中継する手順を整理します

📍 掲載方針

実ドメイン、送信先メールアドレス、SMTPユーザー名、SMTPパスワードは記載せず、再利用できる手順として一般化しています

作業概要

項目 内容
目的 Header FromとMAIL FROMドメインを整合させ、DMARCアライメント、到達性、迷惑メール判定回避の観点を改善します
対象 Amazon SESの検証済みドメインID、Route 53 Hosted Zone、Postfixを利用するLinuxサーバー
作業内容 カスタムMAIL FROM設定、MX/TXTレコード登録、DNS反映確認、PostfixのSES SMTPリレー設定、送信テスト
ℹ️ 情報

Amazon SESでは、通常はSES既定のMAIL FROMドメインが使用されます。独自ドメインのサブドメインをMAIL FROMに利用する場合は、MXレコードとSPF用TXTレコードをDNSへ登録します

置き換え値

値 説明 例
<DOMAIN_NAME>SESで検証済みの送信ドメインexample.com
<MAIL_FROM_SUBDOMAIN>カスタムMAIL FROMに使うサブドメイン名amazon-ses
<REGION>Amazon SESを利用するリージョンap-northeast-1
<SES_SMTP_USERNAME>SES SMTP用のユーザー名SESコンソールで作成
<SES_SMTP_PASSWORD>SES SMTP用のパスワード認証情報管理ルールに従って保管
<RECIPIENT_EMAIL>送信テストの宛先メールアドレス検証済みメールアドレスなど

作業前確認

  1. Amazon SESの対象Identityが Verified であることを確認します
  2. Route 53のHosted Zoneが対象ドメインを管理していることを確認します
  3. 作業前のSES Identity設定、Route 53レコード一覧、DNS応答結果をエビデンスとして取得します
  4. テストメールを送信し、変更前のReceived-SPFやReturn-Pathを確認します

作業前にMXレコードとTXTレコードが存在しないことを確認します

shell
dig MX <MAIL_FROM_SUBDOMAIN>.<DOMAIN_NAME>
dig TXT <MAIL_FROM_SUBDOMAIN>.<DOMAIN_NAME>
powershell
nslookup -type=mx <MAIL_FROM_SUBDOMAIN>.<DOMAIN_NAME>
nslookup -type=txt <MAIL_FROM_SUBDOMAIN>.<DOMAIN_NAME>

カスタムMAIL FROM設定

  1. Amazon SESコンソールを開き、対象Identityを選択します
  2. Custom MAIL FROM domainの編集画面を開きます
  3. カスタムMAIL FROMドメインの使用を有効化します
  4. MAIL FROM domainに <MAIL_FROM_SUBDOMAIN> を入力します
  5. MX障害時の動作を選択し、設定を保存します
⚠️ 注意

MAIL FROMドメインは、SESで検証済みの親ドメイン配下のサブドメインにします。メール送信用や受信用として既に使っているサブドメインは避けます

公式情報: Using a custom MAIL FROM domain - Amazon SES

DNSレコード登録

SESが提示するMXレコードとTXTレコードをRoute 53へ登録します。Route 53を同一AWSアカウントで利用している場合は、SESコンソールからRoute 53へレコードを発行できます

種類 名前 値
MX <MAIL_FROM_SUBDOMAIN>.<DOMAIN_NAME> 10 feedback-smtp.<REGION>.amazonses.com
TXT <MAIL_FROM_SUBDOMAIN>.<DOMAIN_NAME> "v=spf1 include:amazonses.com ~all"
⚠️ 注意

カスタムMAIL FROMドメインには、SES用のMXレコードを1つだけ登録します。複数のMXレコードがあると、SES側の検証に失敗する場合があります

DNS反映確認

Route 53登録後、MXレコードとTXTレコードが名前解決できることを確認します

shell
dig MX <MAIL_FROM_SUBDOMAIN>.<DOMAIN_NAME>
dig TXT <MAIL_FROM_SUBDOMAIN>.<DOMAIN_NAME>

Amazon SESコンソールでは、Custom MAIL FROM domainの状態が Success または検証済み相当の表示になっていることを確認します

SMTP認証情報

PostfixからSES SMTPエンドポイントへ送信する場合は、SES SMTP用の認証情報を作成します

⚠️ 注意

SES SMTPのユーザー名とパスワードは、通常のAWSアクセスキーとは別物です。SMTPパスワードはAWSシークレットアクセスキーではないため、SES SMTP用として発行した値を利用します

公式情報: Obtaining Amazon SES SMTP credentials - Amazon SES

PostfixからSES SMTPへリレー

LinuxサーバーからSES経由でメール送信するため、Postfix、SASL認証パッケージ、メール送信テスト用コマンドを導入します

公式情報: Integrating Amazon SES with Postfix - Amazon SES

root shell
dnf install -y postfix cyrus-sasl-plain mailx

Postfix設定ファイルを退避します

root shell
cp -pi /etc/postfix/main.cf /etc/postfix/main.cf.bak
vi /etc/postfix/main.cf

SES SMTPエンドポイントをリレー先に設定します

main.cf
relayhost = [email-smtp.<REGION>.amazonaws.com]:587
smtp_sasl_auth_enable = yes
smtp_sasl_password_maps = hash:/etc/postfix/sasl_passwd
smtp_sasl_security_options = noanonymous
smtp_tls_security_level = encrypt
smtp_tls_note_starttls_offer = yes
smtp_tls_CAfile = /etc/ssl/certs/ca-bundle.crt

SMTP認証情報ファイルを作成します

root shell
vi /etc/postfix/sasl_passwd
sasl_passwd
[email-smtp.<REGION>.amazonaws.com]:587 <SES_SMTP_USERNAME>:<SES_SMTP_PASSWORD>

認証情報ファイルの権限を制限し、Postfixが参照するハッシュDBを作成します

root shell
chown root:root /etc/postfix/sasl_passwd
chmod 600 /etc/postfix/sasl_passwd
postmap hash:/etc/postfix/sasl_passwd
chown root:root /etc/postfix/sasl_passwd.db
chmod 600 /etc/postfix/sasl_passwd.db

Postfixを再起動し、状態を確認します

root shell
systemctl restart postfix.service
systemctl status postfix.service
ps aux | grep postfix

送信テスト

テストメールを送信します。SESアカウントがサンドボックスの場合は、検証済みの宛先に対して実施します

root shell
mail -s "test mail" <RECIPIENT_EMAIL>

送信ログを確認します

root shell
tail -30 /var/log/maillog

受信メールのソースを開き、Received-SPFやReturn-Pathに <MAIL_FROM_SUBDOMAIN>.<DOMAIN_NAME> が反映されていることを確認します

確認チェックリスト

タイミング 確認内容
作業前SES Identity、Route 53レコード、DNS応答、変更前テストメールのヘッダー
設定後Custom MAIL FROM domainの状態、MX/TXTレコード、DNS反映、Postfix設定差分
送信後メール到達、Postfixログ、SES側の送信結果、Received-SPF、Return-Path
切り戻しカスタムMAIL FROM設定解除、Route 53レコード削除、Postfix設定退避ファイルから復旧
