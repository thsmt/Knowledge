Apache HTTP Server on RHEL 8 / 9

Red Hat Enterprise Linux 8 / 9 で Apache HTTP Server を構築・運用するための実務ガイドです。OS 標準パッケージによる導入から VirtualHost、TLS、SELinux、認証、リバースプロキシ、日常運用までを整理します。

⚠️ 運用の原則

本番設定は /etc/httpd/conf.d/ に個別ファイルとして配置し、変更ごとに apachectl configtest を実行してから再読み込みます。.htaccess は性能低下や意図しない設定変更につながるため、共有ホスティングなど明確な理由がある場合を除き使用しません。

ℹ️ サンプルドメイン

本ページでは公開名を www.example.com、別名を example.com とします。example.com は文書用に予約されたドメインです。実環境では DNS、証明書の SAN、ServerName、監視対象を実際の FQDN へ置き換えます。

対象と基本構成

項目内容
対象 OSRed Hat Enterprise Linux 8 / 9
基本パッケージhttpd

追加設定/etc/httpd/conf.d/*.conf

Apache は Web コンテンツの配信だけでなく、TLS の終端、認証、アプリケーションへのリバースプロキシとして使われます。担当時には「どの VirtualHost がどのドメイン・証明書・公開パス・バックエンドを持つか」を説明できる状態にします。

Apache と NGINX の使い分け

観点Apache HTTP ServerNGINX
代表的な用途既存 Web システム、細かなアクセス制御、各種認証モジュール、アプリケーション連携静的コンテンツ配信、リバースプロキシ、キャッシュ、ロードバランサー
処理モデルMPM を選択し、プロセスまたはスレッドで処理しますイベント駆動で多数の接続を効率よく処理します
設定ディレクトリ単位の設定も可能ですが、本番では中央管理を基本とします設定ファイルを中央管理し、nginx -t 後に再読み込みます
組み合わせNGINX を前段の TLS 終端・プロキシ、Apache をバックエンドとして構成できます。運用主体、監視、ログ、障害時の責任分界を明確にします

製品名だけで優劣を決めず、既存構成、チームの運用経験、必要な認証方式、同時接続数、ベンダーサポートを基準に選択します。

導入と初期確認

1. 基本パッケージを導入し、サービスを起動します

shell
sudo dnf install httpd
sudo systemctl enable --now httpd.service
sudo systemctl status httpd.service --no-pager
httpd -v
sudo httpd -M

2. 必要な通信だけを許可します

firewalld
sudo firewall-cmd --permanent --add-service=http
sudo firewall-cmd --permanent --add-service=https
sudo firewall-cmd --reload
sudo firewall-cmd --list-services

3. ローカルで応答を確認します

shell
curl -I http://127.0.0.1/
sudo ss -lntp | grep -E ':(80|443)'
sudo httpd -S

httpd -S は読み込まれた VirtualHost と ServerName の対応を表示します。意図しない既定 VirtualHost が応答しているときの最初の確認手段です。

設定変更の基本手順

主設定ファイルを直接編集するより、用途ごとに /etc/httpd/conf.d/ 配下へ設定を分けます。変更対象、適用日時、ロールバック手順を残します。

設定ファイル役割
/etc/httpd/conf/httpd.conf待受ポート、実行ユーザー、標準の DocumentRoot などの主設定
/etc/httpd/conf.d/*.confVirtualHost や機能別の追加設定。ファイル名の昇順で読み込まれます
/etc/httpd/conf.d/autoindex.confディレクトリ一覧表示で使用するアイコンや別名の設定
/etc/httpd/conf.d/ssl.confmod_ssl 導入時の TLS 設定
/etc/httpd/conf.d/userdir.confユーザーのホームディレクトリ公開設定
/etc/httpd/conf.d/welcome.conf標準テストページの表示設定
/etc/httpd/conf.modules.d/*.confDSO モジュールの読み込み設定
/etc/httpd/conf/magic必要時に MIME type を判定するための補助設定
shell
sudo install -d -m 0700 /root/config-backup
sudo cp -a /etc/httpd /root/config-backup/httpd-$(date +%Y%m%d)
sudo vi /etc/httpd/conf.d/example.conf
sudo apachectl configtest
sudo systemctl reload httpd.service
sudo systemctl status httpd.service --no-pager
ℹ️ reload と restart

通常の設定変更は既存接続への影響を小さくできる reload を使います。モジュールやプロセスの状態を含めて再起動が必要な場合だけ、影響を確認して restart を選びます。構文エラーがある設定を適用しないため、configtest は省略しません。

主設定または共通設定には、運用窓口として ServerAdmin web-admin@example.com を設定します。設定値はエラーページに表示される場合があるため、個人アドレスではなく監視・運用用の共有アドレスを使います。

VirtualHost の最小構成

/etc/httpd/conf.d/www.example.com.conf
<VirtualHost *:80>
  ServerName www.example.com
  ServerAlias example.com
  ServerAdmin web-admin@example.com
  DocumentRoot /var/www/example

  ErrorLog logs/www-example-error.log
  CustomLog logs/www-example-access.log combined

  <Directory "/var/www/example">
    AllowOverride None
    Options -Indexes +FollowSymLinks
    Require all granted
  </Directory>
</VirtualHost>

ServerName、ServerAlias、DocumentRoot、アクセスログ、エラーログを VirtualHost ごとに明示します。Options -Indexes により、インデックスファイルがないディレクトリの一覧表示を防ぎます。VirtualHost を追加しても DNS レコードは作成されないため、DNS と疎通経路は別に準備します。

content preparation
sudo install -d -m 0755 -o root -g root /var/www/example
sudo vi /var/www/example/index.html
sudo restorecon -Rv /var/www/example
curl -I -H 'Host: www.example.com' http://127.0.0.1/

公開コンテンツの書き込み権限は、デプロイに必要なアカウントだけへ付与します。独自コンテンツを配置した後は、RHEL の標準テストページが外部へ表示されていないことも確認します。

TLS と証明書

TLS 終端を Apache で行う場合、証明書の有効期限、秘密鍵の権限、更新後の再読み込みまでを運用項目にします。暗号スイートやプロトコルは個別に古い値を固定せず、組織の基準と RHEL の暗号ポリシーに従います。

1. mod_ssl を導入し、秘密鍵と CSR を作成します

shell
sudo dnf install mod_ssl
sudo install -d -m 0700 /etc/pki/tls/private
sudo openssl genpkey -algorithm RSA \
  -pkeyopt rsa_keygen_bits:3072 \
  -out /etc/pki/tls/private/www.example.com.key
sudo chmod 0600 /etc/pki/tls/private/www.example.com.key

sudo openssl req -new \
  -key /etc/pki/tls/private/www.example.com.key \
  -subj "/CN=www.example.com" \
  -addext "subjectAltName=DNS:www.example.com,DNS:example.com" \
  -out /etc/pki/tls/certs/www.example.com.csr

openssl req -in /etc/pki/tls/certs/www.example.com.csr -noout -text

CSR を認証局へ提出し、発行されたサーバー証明書と中間証明書を配置します。秘密鍵は外部送付せず、バックアップ時も暗号化とアクセス制御を行います。自己署名証明書は検証用途に限定します。

2. HTTP を HTTPS の正規 URL へ転送します

HTTP VirtualHost
<VirtualHost *:80>
  ServerName www.example.com
  ServerAlias example.com
  Redirect permanent / https://www.example.com/
</VirtualHost>

HTTPS 化する場合は、前項のポート 80 の VirtualHost をこのリダイレクト設定へ置き換えます。example.com への HTTPS 接続も受ける場合、証明書の SAN に両方の名前を含めます。

3. HTTPS の VirtualHost を設定します

/etc/httpd/conf.d/www.example.com-ssl.conf
<VirtualHost *:443>
  ServerName www.example.com
  ServerAlias example.com
  ServerAdmin web-admin@example.com
  DocumentRoot /var/www/example

  SSLEngine on
  SSLCertificateFile /etc/pki/tls/certs/www.example.com.crt
  SSLCertificateKeyFile /etc/pki/tls/private/www.example.com.key

  ErrorLog logs/www-example-ssl-error.log
  CustomLog logs/www-example-ssl-access.log combined
</VirtualHost>

www.example.com を正規 URL として厳密に統一する場合は、example.com 専用のポート 443 VirtualHost を分け、同じ SAN 証明書を設定したうえで www.example.com へ転送します。

4. 設定、証明書、外部接続を確認します

verification
sudo apachectl configtest
sudo systemctl reload httpd.service
openssl s_client -connect www.example.com:443 -servername www.example.com < /dev/null
openssl x509 -in /etc/pki/tls/certs/www.example.com.crt -noout -subject -issuer -enddate

証明書更新は期限前に検証環境で試し、本番では秘密鍵との組み合わせ、SAN、有効期限、チェーン、構文、再読み込み、外部からの接続確認までを一連の作業にします。

SELinux とファイル権限

SELinux を無効化して解決する運用は行いません。403、バックエンド接続失敗、独自ポートでの起動不可は、Linux 権限だけでなく SELinux のラベルとブール値を確認します。

標準外の公開パス

shell
sudo dnf install policycoreutils-python-utils
sudo semanage fcontext -a -t httpd_sys_content_t "/srv/www/example(/.*)?"
sudo restorecon -Rv /srv/www/example

Apache から書き込むパス

shell
sudo semanage fcontext -a -t httpd_sys_rw_content_t "/srv/www/example/uploads(/.*)?"
sudo restorecon -Rv /srv/www/example/uploads

独自ポートとプロキシ接続

shell
sudo semanage port -a -t http_port_t -p tcp 8080
sudo setsebool -P httpd_can_network_connect on
getsebool httpd_can_network_connect

ブール値は Apache がリバースプロキシとして外部接続を開始する場合にだけ必要です。実施前に対象バックエンドと通信先を確認します。拒否ログは ausearch -m AVC -ts recent で確認します。

アクセス制御と認証

管理画面や限定公開コンテンツには、ネットワーク制御と認証を組み合わせます。Basic 認証を使う場合は必ず HTTPS を前提とし、パスワードファイルを公開ディレクトリに置きません。

shell
sudo dnf install httpd-tools
sudo htpasswd -c /etc/httpd/conf/.htpasswd t_hashimoto
sudo chown root:apache /etc/httpd/conf/.htpasswd
sudo chmod 0640 /etc/httpd/conf/.htpasswd
authentication example
<Location "/admin">
  AuthType Basic
  AuthName "Restricted Area"
  AuthUserFile /etc/httpd/conf/.htpasswd
  Require user t_hashimoto
</Location>

<Location "/internal">
  Require ip 192.0.2.0/24
</Location>

組織の ID 管理がある環境では、LDAP、Kerberos、OIDC/SAML 連携を独自実装せず、認証基盤の方式と責任分界を確認します。退職・異動時の権限削除、監査ログ、障害時の迂回手順も設計に含めます。

ユーザーのホームディレクトリを公開する場合

RHEL の既定では UserDir disabled です。https://www.example.com/~t_hashimoto/ のような公開が業務要件にある場合だけ、対象ユーザー、Linux 権限、SELinux、配置物の管理責任を決めて有効化します。

/etc/httpd/conf.d/userdir.conf
UserDir disabled
UserDir enabled t_hashimoto
UserDir public_html

<Directory "/home/*/public_html">
  AllowOverride None
  Options -Indexes
  Require all granted
</Directory>
shell
sudo install -d -m 0755 -o t_hashimoto -g t_hashimoto \
  /home/t_hashimoto/public_html
sudo chmod 0711 /home/t_hashimoto
sudo setsebool -P httpd_enable_homedirs on
sudo restorecon -Rv /home/t_hashimoto/public_html

ホームディレクトリの検索権限を Apache に与えるため、利用者情報や非公開ファイルの配置ルールを確認します。同じオリジン上のユーザーコンテンツは相互に影響する可能性があるため、信頼できない利用者へは提供しません。不要になったら設定と SELinux ブール値を戻します。

リバースプロキシ

Apache をアプリケーションの前段に置く場合は、バックエンドのアドレス、ヘルスチェック、タイムアウト、障害時の応答を決めます。転送プロキシは不要な限り有効化しません。

/etc/httpd/conf.d/app-proxy.conf
ProxyRequests Off
ProxyPreserveHost On

ProxyPass        /app/ http://127.0.0.1:8080/
ProxyPassReverse /app/ http://127.0.0.1:8080/
verification
sudo httpd -M | grep -E 'proxy_module|proxy_http_module'
sudo apachectl configtest
sudo systemctl reload httpd.service
curl -I http://127.0.0.1/app/
⚠️ ProxyRequests は Off のままにします

ProxyRequests On は転送プロキシを有効にします。アクセス制御なしで有効化するとオープンプロキシになるため、リバースプロキシ用途では ProxyPass と ProxyPassReverse を使い、ProxyRequests Off を維持します。

複数バックエンドへ振り分ける場合

単一バックエンド向けの ProxyPass / ProxyPassReverse を、次の構成へ置き換えます。

balancer example
<Proxy "balancer://appcluster">
  BalancerMember "http://192.0.2.10:8080" route=app1
  BalancerMember "http://192.0.2.11:8080" route=app2
  ProxySet lbmethod=byrequests
</Proxy>

ProxyPass        /app/ balancer://appcluster/
ProxyPassReverse /app/ balancer://appcluster/

byrequests はリクエスト数を基準に振り分けます。実運用ではヘルスチェック、接続・応答タイムアウト、セッション維持、切り離し手順、全台障害時の応答を決めます。クライアント IP を扱う場合は X-Forwarded-For を無条件に信用せず、信頼するプロキシの範囲を限定します。

mod_status の安全な公開

status example
<Location "/server-status">
  SetHandler server-status
  Require ip 127.0.0.1 192.0.2.0/24
</Location>

/server-status は要求処理や接続状況を表示するため、インターネットへ公開しません。監視基盤または管理ネットワークに限定します。

チューニング前の確認

  • 負荷試験や実測値で、CPU、メモリ、同時接続数、応答時間、5xx を確認します。
  • MPM は同時に一つだけ有効です。利用中の MPM とプロセス・スレッド上限を把握してから変更します。
  • KeepAlive、Timeout、ワーカー数に固定の推奨値はありません。アプリケーション、接続パターン、サーバー容量に合わせて検証します。
  • 圧縮、キャッシュ、負荷分散は、対象コンテンツ、キャッシュ無効化、セッション維持、障害時の動作を決めてから導入します。

最低限のセキュリティ設定

/etc/httpd/conf.d/security.conf
ServerTokens Prod
ServerSignature Off
TraceEnable Off

<Directory "/var/www/html">
  Options -Indexes +FollowSymLinks
  AllowOverride None
  Require all granted
</Directory>

バージョン情報を抑制しても脆弱性対策にはなりません。OS と httpd の更新、不要モジュールの無効化、管理画面のアクセス制御、証明書更新、ログ監視を継続します。WAF やレート制限は Apache 単体で抱えず、ロードバランサー、CDN、WAF の責任範囲と合わせて設計します。
