[構築] Windows Server 2022

ℹ️ 前提 実環境名、IPアドレス、顧客固有値は扱わず、再利用できる手順として一般化しています。<WINDOWS_PRIVATE_IP>、<HOSTNAME>、<ADMIN_USER>などは環境に合わせて置き換える

0. 起動時の確認

0.1 AMIとキーペア

AWSが提供するWindows Server 2022 AMIを選択し、秘密鍵を安全に管理できるRSAキーペアを指定する

⚠️ 注意 Amazon EC2のWindowsインスタンスはED25519キーペアに対応していません。Administratorの初期パスワードを復号できるよう、Windows ServerではRSAキーペアを使用する

0.2 構築値

値用途
<WINDOWS_PRIVATE_IP>RDP接続先のプライベートIPアドレス
<KEY_PAIR_NAME>Administrator初期パスワードの復号に使用するRSAキーペア
<HOSTNAME>Windows Serverへ設定するコンピューター名
<ADMIN_USER>管理用ローカルユーザー名
<CUSTOMER_USER>利用者用ローカルユーザー名
<VOLUME_DRIVE_LETTER>追加ボリュームへ割り当てるドライブ文字。例: D:

1. Windowsインスタンス接続

1.1 Administratorパスワード取得

  1. Amazon EC2コンソールで対象インスタンスを選択
  2. 「アクション」から「セキュリティ」、「Windowsパスワードを取得」を選択
  3. 起動時に指定したRSAキーペアの秘密鍵をアップロード
  4. 「パスワードを復号化」を選択
  5. 表示されたAdministratorパスワードを認証情報管理ルールに従って保管

1.2 踏み台経由のRDP接続

  1. 踏み台サーバーへSSH接続
  2. SSHクライアントでローカルポートフォワードを設定
  3. ローカルポートへ未使用の任意ポートを指定。例: 65001
  4. 転送先へ<WINDOWS_PRIVATE_IP>:3389を指定
  5. リモートデスクトップクライアントで127.0.0.1:65001へ接続
  6. Administratorと復号したパスワードでサインイン

1.3 Fleet Manager Remote Desktop接続

  1. AWS Systems ManagerのFleet Managerで対象のマネージドノードを選択
  2. 「ノードアクション」から「リモートデスクトップで接続」を選択
  3. ユーザー認証情報を使用する接続方式を選択
  4. Administratorと復号したパスワードで接続
ℹ️ 情報 Fleet Managerを利用するEC2インスタンスには、SSM AgentとAmazonSSMManagedInstanceCore相当の権限を持つIAMインスタンスプロファイルが必要です。インスタンスへアクセスキーを保存せず、IAMロールで権限を付与する

2. ローカルユーザー作成

2.1 管理用ローカルユーザー

  1. Windows + Rを押し、compmgmt.mscを実行
  2. 「ローカル ユーザーとグループ」から「ユーザー」を開く
  3. 「新しいユーザー」を選択
  4. ユーザー名へ<ADMIN_USER>を入力
  5. パスワード運用要件に合わせて初回変更と有効期限を設定
  6. 対象ユーザーをAdministratorsグループへ追加

2.2 利用者用ローカルユーザー

管理者権限でPowerShellを起動し、パスワードを対話入力で受け取る

powershell
$Password = Read-Host "Password for <CUSTOMER_USER>" -AsSecureString
New-LocalUser -Name "<CUSTOMER_USER>" -Password $Password -Description "<CUSTOMER_USER>"

業務上必要な最小権限のローカルグループへ追加する

powershell
Add-LocalGroupMember -Group "Users" -Member "<CUSTOMER_USER>"
⚠️ 注意 Administratorsグループへの追加とパスワード無期限化は、対象ユーザーに管理権限が必要で、社内標準で許可されている場合に限ります。Active Directoryへ参加する環境では、ローカルアカウントを増やさず、ドメインアカウントとグループポリシーによる管理を優先する

3. Windows OS初期設定

3.1 ファイル名拡張子の表示

  1. エクスプローラーを開く
  2. 「表示」から「表示」を開く
  3. 「ファイル名拡張子」を有効化

実行ファイルや設定ファイルの種類を誤認しないよう、拡張子を常に表示する

3.2 コンピューター名設定

  1. Windows + Rを押し、sysdm.cplを実行
  2. 「コンピューター名」タブから「変更」を選択
  3. コンピューター名へ<HOSTNAME>を入力
  4. ドメイン参加が必要な場合は、名前解決と接続先Directory Serviceを確認して設定
  5. 再起動要求を保留し、他の再起動を伴う設定とまとめて実施

3.3 タイムゾーン設定

  1. Windows + Rを押し、timedate.cplを実行
  2. 「タイムゾーンの変更」を選択
  3. (UTC+09:00) 大阪、札幌、東京を選択
ℹ️ 情報 2018年8月以降のAWS提供Windows AMIは、Amazon Time Sync Serviceを既定で使用します。Windows Server 2022の標準AMIでは追加のNTP設定は不要です。ドメイン参加後は、Active Directoryの時刻階層を含む設計に従う

3.4 Windows Update自動更新の無効化

計画したメンテナンスウィンドウで更新を実施する運用に合わせ、自動更新を無効化する

  1. Windows + Rを押し、gpedit.mscを実行
  2. 「コンピューターの構成」、「管理用テンプレート」、「Windowsコンポーネント」、「Windows Update」の順に開く
  3. 「自動更新を構成する」を開く
  4. 「無効」を選択して保存
⚠️ 注意 自動更新を無効化する場合は、AWS Systems Manager Patch Manager、WSUS、SCCMなどで定期的に更新を適用する運用を必ず用意します。更新経路がない状態で無効化すると、既知の脆弱性が残る

3.5 RDPセキュリティ設定

3.5.1 ネットワークレベル認証

  1. Windows + Rを押し、sysdm.cplを実行
  2. 「リモート」タブを開く
  3. 「ネットワーク レベル認証でリモート デスクトップを実行しているコンピューターからのみ接続を許可する」を有効化

ネットワークレベル認証は、RDPセッションを確立する前に利用者を認証し、未認証接続によるリソース消費と攻撃対象領域を抑える

Microsoft公式: Remote Desktopとネットワークレベル認証

3.5.2 RDPセッション制限

  1. Windows + Rを押し、gpedit.mscを実行
  2. 「管理用テンプレート」、「Windowsコンポーネント」、「リモート デスクトップ サービス」、「リモート デスクトップ セッション ホスト」、「接続」の順に開く
  3. 「接続数を制限する」を開き、運用要件に合わせて設定
  4. 複数セッションが必要な場合のみ、「リモート デスクトップ サービス ユーザーに対してリモート デスクトップ サービス セッションを1つに制限する」を無効化

3.5.3 RDP暗号化レベル

RDPのネイティブ暗号化を利用する接続に対して、クライアントとサーバー間の暗号化レベルを高へ固定する

  1. Windows + Rを押し、gpedit.mscを実行
  2. 「コンピューターの構成」、「管理用テンプレート」、「Windowsコンポーネント」の順に展開
  3. 「リモート デスクトップ サービス」、「リモート デスクトップ セッション ホスト」、「セキュリティ」の順に展開
  4. 「クライアント接続の暗号化レベルを設定する」を開く
  5. 「有効」を選択し、暗号化レベルへ「高」を指定
  6. gpupdate /forceを実行
ℹ️ 情報 「高」はRDPのネイティブ暗号化へ128ビット暗号化を要求する設定です。TLSをセキュリティレイヤーとして使用する接続では、証明書とTLS設定を優先して設計します。FIPS準拠設定は互換性へ影響するため、準拠要件がある場合だけ別途検証する

Microsoft公式: Remote Desktop Servicesポリシー

3.6 Windows Defender Firewallの無効化

本手順の対象環境では、通信制御をSecurity Groupなどの上位レイヤーへ集約し、OS側との二重管理による設定差分や疎通障害を避ける社内標準に基づいてWindows Defender Firewallを無効化する

  1. Windows + Rを押し、wf.mscを実行
  2. 「Windows Defender ファイアウォールのプロパティ」を開く
  3. ドメイン、プライベート、パブリックの各プロファイルで「ファイアウォールの状態」を「無効」に変更
  4. 設定を保存
⚠️ 注意 Windows Defender Firewallの無効化は、すべての環境に適用する一般的なAWS推奨ではありません。AWSはOS側ファイアウォールを多層防御として利用することを推奨しています。本手順は、Security GroupとNetwork ACLによる通信制御、EDRによる検知・防御、変更承認が整備された対象環境に限って適用する

3.7 追加ボリュームの初期化

  1. Windows + Rを押し、diskmgmt.mscを実行
  2. 未初期化ディスクの容量とディスク番号を確認
  3. パーティションスタイルへGPTを選択
  4. 未割り当て領域から「新しいシンプル ボリューム」を選択
  5. ドライブ文字へ<VOLUME_DRIVE_LETTER>を割り当て
  6. ファイルシステムへNTFSを選択し、要件に合わせたボリュームラベルを設定
  7. クイックフォーマットを実行

3.8 Windows Update

  1. Windows + Rを押し、ms-settings:windowsupdateを実行
  2. 「更新プログラムのチェック」を選択
  3. 検出された更新プログラムをインストール
  4. 再起動要求がある場合は、作業影響を確認して再起動
  5. 更新プログラムが検出されなくなるまで繰り返し実行

4. AWS管理機能の設定

4.1 CloudWatch Agent導入

AWS Systems Manager Run Commandを使用して、Amazon CloudWatch Agentをインストールします。インストールだけではメトリクスやログは収集されないため、後続手順で設定ファイルを作成して適用する

  1. AWS Systems Managerコンソールで「Run Command」を開く
  2. コマンドドキュメントへAWS-ConfigureAWSPackageを選択
  3. ActionへInstall、NameへAmazonCloudWatchAgentを指定
  4. 対象インスタンスをタグまたはインスタンスIDで指定
  5. コマンドを実行
ℹ️ 情報 EC2インスタンスのIAMロールには、Systems Manager管理用のAmazonSSMManagedInstanceCoreと、メトリクス・ログ送信用のCloudWatchAgentServerPolicy相当の権限を付与します。設定ファイルにアクセスキーやシークレットアクセスキーを保存しない

AWS公式: CloudWatch Agentのインストール

4.2 CloudWatch Agent設定ファイル作成

収集するWindowsイベントログ、パフォーマンスカウンター、収集間隔、CloudWatch Logsのロググループ名をJSON形式の設定ファイルへ定義する

分類設定例設計時の確認事項
WindowsイベントログSystem、Application収集するイベントレベル、保持期間、機密情報の有無
WindowsセキュリティログSecurity監査要件がある場合のみ追加し、ログ量と閲覧権限を確認
CPUProcessor(_Total)\% Processor Time収集間隔とアラームしきい値
メモリMemory\% Committed Bytes In UseOS標準メトリクスでは取得できないため、Agentで収集
ディスクLogicalDisk(*)\% Free Space一時ドライブや不要なボリュームを対象外にするか確認

4.2.1 Windows上のウィザードで作成

単体サーバーで設定内容を確認しながら作成する場合は、管理者権限のPowerShellから設定ウィザードを実行する

powershell
Set-Location "$Env:ProgramFiles\Amazon\AmazonCloudWatchAgent"
.\amazon-cloudwatch-agent-config-wizard.exe
  1. OSへWindows、実行環境へEC2を指定
  2. 収集間隔とパフォーマンスカウンターを選択
  3. 収集するWindowsイベントログ、イベントレベル、ロググループ名を指定
  4. ローカル利用の場合は、生成されたconfig.jsonを確認
  5. Parameter Storeで集中管理する場合は、保存確認へYesを選択し、パラメータ名とリージョンを指定

ローカルのconfig.jsonを使用する場合は、次のコマンドで設定を読み込み、Agentを起動する

powershell
& "$Env:ProgramFiles\Amazon\AmazonCloudWatchAgent\amazon-cloudwatch-agent-ctl.ps1" `
  -a fetch-config -m ec2 -s `
  -c "file:$Env:ProgramFiles\Amazon\AmazonCloudWatchAgent\config.json"

4.2.2 Parameter Storeへ保存

複数台へ同じ設定を配布する場合は、承認済みのJSONファイルをAWS Systems Manager Parameter Storeへ保存します。AWS管理ポリシーのCloudWatchAgentServerPolicyを利用する場合は、AmazonCloudWatch-WindowsServer2022のようにAmazonCloudWatch-で始まるパラメータ名を指定する

powershell
aws ssm put-parameter `
  --name "<CLOUDWATCH_AGENT_CONFIG_PARAMETER>" `
  --type "String" `
  --value file://cloudwatch-agent.json `
  --overwrite
⚠️ 注意 Parameter Storeへ設定を登録する作業者にはssm:PutParameter権限が必要です。EC2インスタンスにはssm:GetParameterだけを許可し、設定登録権限は付与しません。AmazonCloudWatch-以外のパラメータ名を使う場合は、対象パラメータに限定したssm:GetParameter権限をインスタンスロールへ追加します。Parameter Storeはアカウントおよびリージョン単位で管理されるため、適用先と同じリージョンへ登録する

AWS公式: ウィザードによる設定ファイル作成

4.3 Run Commandによる設定適用

Parameter Storeに保存した設定を、AWS Systems Manager Run Commandから対象インスタンスへ適用する

  1. AWS Systems Managerコンソールで「Run Command」を開く
  2. コマンドドキュメントへAmazonCloudWatch-ManageAgentを選択
  3. Actionへconfigureを指定
  4. Modeへec2を指定
  5. Optional Configuration Sourceへssmを指定
  6. Optional Configuration Locationへ<CLOUDWATCH_AGENT_CONFIG_PARAMETER>を指定
  7. Optional Restartへyesを指定
  8. 対象インスタンスをタグまたはインスタンスIDで指定し、コマンドを実行
  9. コマンド出力が成功していることを確認し、CloudWatchのメトリクスとロググループへデータが到着していることを確認
ℹ️ 情報 共通設定をParameter Storeで管理すると、インスタンスへJSONファイルを個別配布せずに同じ設定を再適用できます。設定変更時はJSONをレビューし、Parameter Storeを更新してから対象タグ単位でRun Commandを実行する

AWS公式: Systems ManagerによるCloudWatch Agentの設定と起動

4.4 EC2Launch v2とAMI横展開

AWS提供のWindows Server 2022 AMIにはEC2Launch v2が含まれます。構築済みインスタンスをカスタムAMIとして横展開する場合は、EC2Launch v2からSysprepを実行してインスタンス固有情報を一般化する

  1. 構築作業とWindows Updateを完了
  2. 一時ファイル、個別の認証情報、作業用アカウント、固定IP設定などを除去
  3. スタートメニューから「EC2Launch settings」を開く
  4. AMI起動後に必要なタスクとSSM Agentの起動設定を確認
  5. 「Shutdown with Sysprep」を実行
  6. インスタンス停止後にAmazon EC2コンソールからAMIを作成
⚠️ 注意 Sysprepはバックアップ手段ではありません。コンピューター名やSIDなどのインスタンス固有情報を一般化するため、稼働中インスタンスの復旧用AMIを取得する目的では使用しない

AWS公式: EC2Launch v2設定

5. 要件に応じた追加設定

5.1 TLS 1.0 / 1.1の無効化

組織の暗号化基準でTLS 1.2以上を必須とする場合は、IIS、.NET Framework、データベースクライアント、監視エージェントなどの互換性を検証してからTLS 1.0 / 1.1を無効化する

5.1.1 事前確認

  1. 対象サーバーで稼働するIIS、データベースクライアント、監視・バックアップ・セキュリティエージェントを一覧化
  2. 接続先と接続元がTLS 1.2以上に対応していることを確認
  3. 設定変更後に確認するWeb、API、SMTP、データベース接続を決定
  4. 再起動可能な作業時間と切り戻し手順を確認

5.1.2 Schannel設定のバックアップ

管理者権限のPowerShellで、変更前のSchannel設定をエクスポートする

powershell
New-Item -ItemType Directory -Path C:\Work\tls-backup -Force
reg export "HKLM\SYSTEM\CurrentControlSet\Control\SecurityProviders\SCHANNEL" `
  "C:\Work\tls-backup\schannel-before.reg" /y

5.1.3 TLS 1.0 / 1.1の無効化

TLS 1.0とTLS 1.1のClient、ServerへEnabled=0とDisabledByDefault=1を設定する

powershell
$protocols = @('TLS 1.0', 'TLS 1.1')
$roles = @('Client', 'Server')

foreach ($protocol in $protocols) {
  foreach ($role in $roles) {
    $path = "HKLM:\SYSTEM\CurrentControlSet\Control\SecurityProviders\SCHANNEL\Protocols\$protocol\$role"
    New-Item -Path $path -Force | Out-Null
    New-ItemProperty -Path $path -Name Enabled -PropertyType DWord -Value 0 -Force | Out-Null
    New-ItemProperty -Path $path -Name DisabledByDefault -PropertyType DWord -Value 1 -Force | Out-Null
  }
}

設定値を確認する

powershell
Get-ChildItem "HKLM:\SYSTEM\CurrentControlSet\Control\SecurityProviders\SCHANNEL\Protocols" |
  Where-Object PSChildName -in @('TLS 1.0', 'TLS 1.1') |
  ForEach-Object {
    Get-ItemProperty "$($_.PSPath)\Client"
    Get-ItemProperty "$($_.PSPath)\Server"
  } | Select-Object PSPath, Enabled, DisabledByDefault
  1. サーバーを再起動
  2. IIS、API、SMTP、データベース、監視エージェントの通信を確認
  3. Windowsイベントログと対象アプリケーションログでTLSエラーの有無を確認
⚠️ 注意 Schannel設定はOS全体へ影響します。互換性問題が発生した場合はschannel-before.regをreg importで復元し、再起動して切り戻します。Microsoftはレジストリの直接変更より、グループポリシーや管理された構成方式の利用を推奨している

Microsoft公式: WindowsにおけるTLS 1.0 / 1.1の非推奨化

5.2 イベントログとクラッシュダンプ

監視要件と障害解析要件に合わせて、Windowsイベントログの最大サイズと上書き方式、ページファイル、クラッシュダンプの保存方式を設計します。ローカルログだけに依存せず、CloudWatch Logsなどの集中管理先へ転送する

5.2.1 Windowsイベントログ

現在の最大サイズと保持方式を確認する

powershell
wevtutil gl Application
wevtutil gl System
wevtutil gl Security

次の例では、各ログの最大サイズを256MBへ設定し、上限到達時は古いイベントから上書きする

powershell
wevtutil sl Application /ms:268435456 /rt:false
wevtutil sl System /ms:268435456 /rt:false
wevtutil sl Security /ms:268435456 /rt:false
ℹ️ 情報 /rt:falseは上限到達時に古いイベントを上書きします。/rt:trueは既存イベントを保持し、上限到達後の新規イベントを保存しません。監査要件がある場合はログ転送と保管先を先に設計する

5.2.2 ページファイル

  1. Windows + Rを押し、sysdm.cplを実行
  2. 「詳細設定」タブの「パフォーマンス」から「設定」を選択
  3. 「詳細設定」タブの「仮想メモリ」から「変更」を選択
  4. 原則として「すべてのドライブのページング ファイルのサイズを自動的に管理する」を有効化
  5. 完全メモリダンプを取得する場合は、OSボリューム上にメモリ容量とダンプ方式に応じたページファイルを確保

5.2.3 クラッシュダンプ

  1. Windows + Rを押し、sysdm.cplを実行
  2. 「詳細設定」タブの「起動と回復」から「設定」を選択
  3. 「デバッグ情報の書き込み」へ「自動メモリ ダンプ」を選択
  4. ダンプファイルへ%SystemRoot%\MEMORY.DMPを指定
  5. 「既存のファイルに上書きする」を有効化

レジストリ上のダンプ方式と保存先を確認する

powershell
Get-ItemProperty "HKLM:\SYSTEM\CurrentControlSet\Control\CrashControl" |
  Select-Object CrashDumpEnabled, DumpFile, MinidumpDir, AlwaysKeepMemoryDump
CrashDumpEnabledダンプ方式
1完全メモリダンプ
2カーネルメモリダンプ
