CloudFormation 運用手順集

CloudFormationテンプレートの書き方だけでなく、コンソール、S3 URL、Git sync、AWS CLIなどのデプロイ方法を使い分け、変更セット、ロールバック、削除、ドリフト検出までを運用手順として整理します

📍 目的 本ページでは、各リポジトリ固有のテンプレート説明ではなく、CloudFormationを安全に扱うための共通手順、デプロイ方法、確認観点をまとめます

基本方針

  • テンプレートはGitで管理し、環境差分はParametersとパラメータファイルに分離します
  • 機密値はテンプレートやGitに含めず、NoEcho、Secrets Manager、SSM Parameter Storeを検討します
  • 本番更新前は変更セットを作成し、作成・変更・削除・置換されるリソースを確認します
  • スタック削除前は依存リソース、保持すべきデータ、削除保護、課金影響を確認します

デプロイ方法の使い分け

方法 向いている場面 注意点
テンプレートファイルのアップロード 手元のYAML/JSONをすぐに試す検証作業 コンソールがテンプレートをS3へアップロードします。正式版はGitに残します
Amazon S3 URL 同じテンプレートを複数回利用する作業、共有されたテンプレートを使う作業 事前にS3へテンプレートを配置し、読み取り可能なURLを指定します
Git から同期 GitのPull Request、履歴、レビューを使ってスタックを継続更新したい場合 GitHub接続、リポジトリ、ブランチ、デプロイファイル、テンプレートパス、IAMロールを準備します
Infrastructure Composer 構成を視覚的に整理し、初期テンプレートを作る場合 継続運用では、出力したテンプレートをGitにコミットしてレビューする流れに寄せます
AWS CLI 手順を再現したい場合、パラメータファイル込みで定型化したい場合 実行アカウント、リージョン、IAM権限、capabilitiesの指定を事前に確認します

新規作成とリソースインポート

コンソールの選択肢 意味 使いどころ
新しいリソースを使用（標準） テンプレートに定義したAWSリソースをCloudFormationで新規作成します 通常の新規デプロイ、検証環境作成、リポジトリのテンプレート確認
既存のリソースを使用（リソースをインポート） すでに存在するAWSリソースを、CloudFormationスタックの管理対象へ取り込みます 手動作成リソースをIaC管理へ移行したい場合。通常の新規作成では使いません
ℹ️ 情報 リソースインポートでは、既存リソースを表すテンプレートと識別子の指定が必要です。既存リソースを削除して作り直す操作ではありません

テンプレート検証

構文レベルの確認として、validate-template を実行します。リソース間の依存関係やIAM権限までは完全には検出できないため、変更セットや実デプロイ前確認と組み合わせます

powershell 
aws cloudformation validate-template `
  --template-body file://templates/example.yml

パラメータ設計

観点 記載する内容
環境差分 ProjectName、EnvironmentName、CIDR、AZ、InstanceTypeなどをパラメータ化します
入力制約 AllowedValues、AllowedPattern、MinLength、MaxLengthを使い、誤入力を減らします
機密値 DBパスワードなどはNoEchoを設定します。ただしNoEchoだけで完全な秘匿にはならないため、Secrets Managerも検討します
命名 リソース名はProjectNameとEnvironmentNameを組み合わせ、長さ制限に注意します

テンプレートファイルのアップロード

  1. CloudFormationコンソールで スタックの作成 を選択します
  2. 新しいリソースを使用（標準） を選択します
  3. 既存のテンプレートを選択 を選び、テンプレートソースで テンプレートファイルのアップロード を選択します
  4. ローカルのYAMLまたはJSONファイルを選択します
  5. CloudFormationがテンプレートをS3へアップロードし、スタック作成画面に進みます
📍 使いどころ 検証や一時確認では手軽です。一方で、アップロードしたファイルだけでは変更履歴が残りにくいため、最終版はGitリポジトリにコミットします

Amazon S3 URLで作成

S3 URLを使う場合は、事前にテンプレートファイルをS3へ配置し、そのテンプレートを読み取れるURLをCloudFormationに指定します

テンプレートをS3へアップロードします

powershell 
aws s3 cp templates/example.yml `
  s3://<BUCKET_NAME>/cloudformation/templates/example.yml

コンソールの Amazon S3 URL に、以下のようなURLを指定します

text 
https://<BUCKET_NAME>.s3.<REGION>.amazonaws.com/cloudformation/templates/example.yml
ℹ️ 情報 テンプレートがネストスタックを参照する場合は、参照先のテンプレートもS3上に配置します。S3バケットやオブジェクトの権限、リージョン、バージョニング有無も確認します

Gitから同期

Git syncは、CloudFormationテンプレートファイルとスタックデプロイファイルをGitリポジトリで管理し、対象ブランチへの変更をスタック更新に連動させる方法です

項目 指定する内容
接続 GitHubなどのリポジトリプロバイダーとの接続を作成または選択します
リポジトリ / ブランチ CloudFormationが監視するリポジトリとブランチを選択します
デプロイファイルのパス スタック名、テンプレートファイル、パラメータなどを管理するファイルのパスを指定します
テンプレートファイルパス リポジトリ内のCloudFormationテンプレートのパスを指定します
IAMロール CloudFormationがGitリポジトリから同期してスタックを操作するためのロールを指定します
Pull Requestコメント 有効化すると、変更セットの内容をPull Requestにコメントできます

デプロイファイルは、環境ごとに分けると差分が追いやすくなります

text 
templates/prod.yml
deployments/prod-stack.yaml
deployments/stg-stack.yaml
📍 判断基準 Git syncはレビュー履歴を残せる一方で、初回の接続、IAMロール、デプロイファイル設計が必要です。まずは通常のコンソール作成やCLIで動作を確認し、継続運用するテンプレートだけGit sync化するのが扱いやすいです

Infrastructure Composerの扱い

Infrastructure Composerは、AWSリソースを視覚的に配置しながらCloudFormationテンプレートを作成・編集できる機能です。構成を整理する初期段階や学習には便利ですが、リポジトリ運用では出力されたテンプレートをGitで管理し、レビュー、検証、変更セット確認の流れに戻します

使いやすい場面 注意する場面
構成図に近い感覚で初期テンプレートを作る 既存リポジトリの細かい命名規則、パラメータ設計、コメント整理を維持する
リソース同士の関係を視覚的に確認する Lambdaなど、コードやパッケージングが絡む構成をそのまま完結させる

AWS CLIでスタック作成

IAMリソースを作成する場合は CAPABILITY_NAMED_IAM または CAPABILITY_IAM が必要です。必要な場合だけ指定します

powershell 
aws cloudformation create-stack `
  --stack-name sample-stack `
  --template-body file://templates/example.yml `
  --parameters file://parameters/example.json `
  --capabilities CAPABILITY_NAMED_IAM

変更セット

本番更新では、直接 update-stack する前に変更セットを作成し、置換、削除、IAM変更の有無を確認します

powershell 
aws cloudformation create-change-set `
  --stack-name sample-stack `
  --change-set-name update-20260712 `
  --template-body file://templates/example.yml `
  --parameters file://parameters/example.json `
  --capabilities CAPABILITY_NAMED_IAM
powershell 
aws cloudformation describe-change-set `
  --stack-name sample-stack `
  --change-set-name update-20260712

ロールバック・ドリフト検出

ROLLBACK_COMPLETE や UPDATE_ROLLBACK_FAILED は、イベントタブで失敗したリソースと理由を確認します。原因を特定するまでは、同じテンプレートで再実行しないことを基本とします

状態 初動
ROLLBACK_COMPLETE 作成失敗です。イベントを確認し、必要に応じてスタックを削除してから再作成します
UPDATE_ROLLBACK_COMPLETE 更新失敗後に元の状態へ戻っています。差分と失敗理由を確認してから再更新します
UPDATE_ROLLBACK_FAILED 手動変更や削除済みリソースが影響している可能性があります。対象リソースを確認し、continue-update-rollbackを検討します

手動変更の有無を確認する場合は、ドリフト検出を実行します

powershell 
aws cloudformation detect-stack-drift `
  --stack-name sample-stack
