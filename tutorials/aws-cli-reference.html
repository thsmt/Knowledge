AWS CLI コマンドリファレンス

日常運用でよく使うAWS CLIの初期設定、IAM、Security Group、Prefix List、SNSに加え、EBS拡張、WAF IPセット更新、IAMユーザーの利用停止を整理します

📍 使い分け

本ページは、コンソールでも可能だがCLIの方が早い定型操作をまとめます。CLI/APIでしか変更・切り戻し・復旧できない操作は、AWS CLI 特殊対応 に分けています

AWS CLIの初期設定

インストール

作業端末のOSに合わせてAWS CLIをインストールします

powershell
winget install --id Amazon.AWSCLI -e --source winget
aws --version
zsh
brew install awscli
aws --version
bash
curl "https://awscli.amazonaws.com/awscli-exe-linux-x86_64.zip" -o "awscliv2.zip"
unzip awscliv2.zip
sudo ./aws/install
aws --version

認証情報の確認

現在の認証情報が、想定しているAWSアカウントとIAMユーザー/ロールを向いているか確認します

powershell
aws sts get-caller-identity

認証情報の設定

作業端末で初めてAWS CLIを使う場合は、デフォルト認証情報を設定します

powershell
aws configure
ℹ️ 情報 --profile は、複数のAWSアカウントやIAMユーザーを名前付きプロファイルで切り替える場合だけ指定します。このページでは、デフォルト認証情報を使う前提で記載します

リージョン確認

作業前に対象リージョンを明示します。東京リージョンで作業する場合は ap-northeast-1 を指定します

powershell
aws configure set region ap-northeast-1

AWS Identity and Access Management (IAM)

人のログイン用途はAWS IAM Identity Centerの利用を優先します。ここでは、個別要件でIAMユーザー、IAMグループ、IAMポリシー、アクセスキー、コンソールログインプロファイルをCLIで扱う場合の基本操作を整理します。詳細は create-group、attach-group-policy、create-policy、create-access-key、create-login-profile を確認します

値説明
<USER_NAME>作成または確認するIAMユーザー名
<GROUP_NAME>ユーザーを所属させるIAMグループ名
<POLICY_NAME>作成するカスタマー管理ポリシー名
<CUSTOM_POLICY_ARN>作成済みカスタマー管理ポリシーのARN
<TEMPORARY_PASSWORD>初回ログイン用の一時パスワード

IAMユーザーを作成します

powershell
aws iam create-user --user-name <USER_NAME>

IAMグループを作成します

powershell
aws iam create-group --group-name <GROUP_NAME>

作成したIAMユーザーを既存グループに追加します

powershell
aws iam add-user-to-group --user-name <USER_NAME> --group-name <GROUP_NAME>

AWS管理ポリシーをIAMグループにアタッチします。例では読み取り専用の ReadOnlyAccess を使用します

powershell
aws iam attach-group-policy --group-name <GROUP_NAME> --policy-arn arn:aws:iam::aws:policy/ReadOnlyAccess

カスタマー管理ポリシーを作成する場合は、ポリシードキュメントをJSONで用意します

json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "ec2:Describe*"
      ],
      "Resource": "*"
    }
  ]
}

JSONファイルからカスタマー管理ポリシーを作成します

powershell
aws iam create-policy --policy-name <POLICY_NAME> --policy-document file://policy.json

作成したカスタマー管理ポリシーをIAMグループにアタッチします

powershell
aws iam attach-group-policy --group-name <GROUP_NAME> --policy-arn <CUSTOM_POLICY_ARN>

IAMグループにアタッチされているポリシーを確認します

powershell
aws iam list-attached-group-policies --group-name <GROUP_NAME>

ユーザーの所属グループを確認します

powershell
aws iam list-groups-for-user --user-name <USER_NAME>

IAMユーザーのアクセスキーを発行します

powershell
aws iam create-access-key --user-name <USER_NAME>
⚠️ 注意

CLIでアクセスキーを発行した場合、マネジメントコンソールのようなCSVファイルは自動ダウンロードされません

AccessKeyId と SecretAccessKey はコマンド実行結果に一度だけ表示されます。必要な場合は、安全な場所へ即時保管します

IAMユーザーへコンソールログイン用のログインプロファイルを作成します

powershell
aws iam create-login-profile --user-name <USER_NAME> --password "<TEMPORARY_PASSWORD>" --password-reset-required
ℹ️ 情報

CLIでコンソールログインを有効化する場合は、create-login-profile でパスワードを指定します

GUIのように自動生成パスワードをCSVでダウンロードする流れではありません。実運用では、パスワードをコマンド履歴へ残さない運用、初回変更必須、利用期限の管理をセットで扱います

Security Group ルール追加・削除

Security Group変更は影響範囲が広いため、対象SG、方向、プロトコル、ポート、送信元/宛先、説明欄を事前に確定します

AWS CLIのSecurity Groupルールで全プロトコルを指定する場合は、公式リファレンスに合わせて --protocol -1 を使用します

値説明
<SECURITY_GROUP_ID>変更対象のSecurity Group ID
<CIDR>許可または削除する送信元CIDR。例: 203.0.113.10/32
<DESCRIPTION>Security Groupルールの説明。例: office-a temporary access

作業前に、対象Security Groupの現在のルールを確認します

powershell
aws ec2 describe-security-groups --group-ids <SECURITY_GROUP_ID>

HTTPの80/tcpだけを許可する例です

powershell
aws ec2 authorize-security-group-ingress --group-id <SECURITY_GROUP_ID> --protocol tcp --port 80 --cidr <CIDR>

説明欄を付けて80/tcpを許可する例です。ルールの説明はトップレベルの --description ではなく、--ip-permissions の IpRanges 側に指定します

powershell
aws ec2 authorize-security-group-ingress --group-id <SECURITY_GROUP_ID> --ip-permissions 'IpProtocol=tcp,FromPort=80,ToPort=80,IpRanges=[{CidrIp=<CIDR>,Description="<DESCRIPTION>"}]'

全プロトコル・全ポートを許可する例です。検証や一時作業以外では原則として避け、作業後に必ず削除します

powershell
aws ec2 authorize-security-group-ingress --group-id <SECURITY_GROUP_ID> --protocol -1 --cidr <CIDR>

Security GroupルールID、CIDR、説明欄を一覧で確認します

powershell
aws ec2 describe-security-group-rules --filters Name=group-id,Values=<SECURITY_GROUP_ID> --query "SecurityGroupRules[*].[SecurityGroupRuleId,IsEgress,IpProtocol,FromPort,ToPort,CidrIpv4,Description]" --output table

80/tcpの許可ルールを削除する例です

powershell
aws ec2 revoke-security-group-ingress --group-id <SECURITY_GROUP_ID> --protocol tcp --port 80 --cidr <CIDR>

全プロトコル・全ポートの許可ルールを削除する例です

powershell
aws ec2 revoke-security-group-ingress --group-id <SECURITY_GROUP_ID> --protocol -1 --cidr <CIDR>

Prefix List作成・IP登録

接続元IPが複数ある場合は、Security GroupへCIDRを1件ずつ直接追加するのではなく、Customer-managed Prefix ListにまとめてからSecurity Groupで参照します。変更対象をPrefix List側に寄せられるため、接続元IPの追加・削除を運用しやすくなります。詳細は Managed prefix lists、create-managed-prefix-list、modify-managed-prefix-list、authorize-security-group-ingress を確認します

⚠️ 注意 Security Groupの画面上は1つのPrefix List参照として整理できます。ただし、Customer-managed Prefix ListをSecurity Groupで参照する場合、クォータ上はPrefix Listの最大エントリ数分として扱われます。最大エントリ数は余裕を持たせすぎず、運用に必要な範囲で設計します
値説明
<PREFIX_LIST_NAME>作成または更新するPrefix List名
<MAX_ENTRIES>Prefix Listに登録できる最大エントリ数。例: 10
<CIDR_1>登録する接続元CIDR。例: 203.0.113.10/32
<CIDR_2>追加で登録する接続元CIDR。不要な場合はコマンドから削除します
<CIDR_3>既存Prefix Listへ後から追加する接続元CIDR
<DESCRIPTION_1>登録CIDRの説明。例: office-a
<DESCRIPTION_2>追加CIDRの説明
<DESCRIPTION_3>後から追加するCIDRの説明
<PREFIX_LIST_ID>作成後に払い出されるPrefix List ID
<CURRENT_VERSION>Prefix List更新時に指定する現在のバージョン
<SECURITY_GROUP_ID>Prefix Listを参照させるSecurity Group ID
<PORT>許可するポート番号。例: 443

Prefix Listを作成し、初期CIDRを登録します。登録するIPアドレスが複数ある場合は、Cidr=... の指定を追加します

powershell
aws ec2 create-managed-prefix-list --prefix-list-name <PREFIX_LIST_NAME> --address-family IPv4 --max-entries <MAX_ENTRIES> --entries Cidr=<CIDR_1>,Description="<DESCRIPTION_1>" Cidr=<CIDR_2>,Description="<DESCRIPTION_2>"

作成したPrefix ListのID、現在のバージョン、最大エントリ数を確認します

powershell
aws ec2 describe-managed-prefix-lists --filters Name=prefix-list-name,Values=<PREFIX_LIST_NAME> --query "PrefixLists[*].[PrefixListId,PrefixListName,Version,MaxEntries,State]" --output table

既存のPrefix Listへ接続元CIDRを追加します。更新時は、直前に確認した Version を <CURRENT_VERSION> に指定します

powershell
aws ec2 modify-managed-prefix-list --prefix-list-id <PREFIX_LIST_ID> --current-version <CURRENT_VERSION> --add-entries Cidr=<CIDR_3>,Description="<DESCRIPTION_3>"

Prefix Listに登録されているCIDRを確認します

powershell
aws ec2 get-managed-prefix-list-entries --prefix-list-id <PREFIX_LIST_ID> --query "Entries[*].[Cidr,Description]" --output table

Security GroupのインバウンドルールでPrefix Listを参照します。例では、Prefix Listに登録した接続元CIDR群から <PORT> へのTCP通信を許可します

powershell
aws ec2 authorize-security-group-ingress --group-id <SECURITY_GROUP_ID> --ip-permissions IpProtocol=tcp,FromPort=<PORT>,ToPort=<PORT>,PrefixListIds=[{PrefixListId=<PREFIX_LIST_ID>}]

SNS Topic・Subscription作成

通知先を作成する場合は、Topic、Subscription、確認メールの承認、通知テスト、設定確認、不要時の解除までを一連の手順として扱います。詳細は subscribe を確認します

値説明
<TOPIC_NAME>作成するSNS Topic名
<TOPIC_ARN>作成済みSNS TopicのARN
<MAIL_ADDRESS>通知先メールアドレス
<SUBSCRIPTION_ARN>SubscriptionのARN。承認前は PendingConfirmation になります

SNS Topicを作成します

powershell
aws sns create-topic --name <TOPIC_NAME>

SNS Topic一覧を確認します

powershell
aws sns list-topics

メール通知先をSubscriptionとして登録します

powershell
aws sns subscribe --topic-arn <TOPIC_ARN> --protocol email --notification-endpoint <MAIL_ADDRESS>
ℹ️ 情報

email プロトコルのSubscriptionは、通知先に届く確認メールを承認するまで配信されません

承認前はSubscription ARNが PendingConfirmation と表示されます

Topicに紐づくSubscriptionを確認します

powershell
aws sns list-subscriptions-by-topic --topic-arn <TOPIC_ARN>

Topic属性を確認します

powershell
aws sns get-topic-attributes --topic-arn <TOPIC_ARN>

通知テストを実行します

powershell
aws sns publish --topic-arn <TOPIC_ARN> --message "notification test"

不要になったSubscriptionを解除します

powershell
aws sns unsubscribe --subscription-arn <SUBSCRIPTION_ARN>

EC2 / EBS ボリューム拡張

ファイルシステムの空き容量が不足した場合に、EBSボリュームの容量をオンラインで拡張します。EBSは縮小できないため、作業前にスナップショットを取得し、必要なサイズを確定します。OS側のパーティション・ファイルシステム拡張は [構築] Red Hat Enterprise Linux 9 を参照します

値説明
<VOLUME_ID>拡張するEBSボリュームID
<TARGET_SIZE_GIB>拡張後のサイズ（GiB）。現在のサイズ以上を指定

作業前の復旧ポイントとしてスナップショットを作成

powershell
aws ec2 create-snapshot --volume-id <VOLUME_ID> --description "pre-resize backup"

権限と入力値の事前確認。DryRunOperation が返る場合は、実行権限があります

powershell
aws ec2 modify-volume --volume-id <VOLUME_ID> --size <TARGET_SIZE_GIB> --dry-run

EBSボリュームの拡張

powershell
aws ec2 modify-volume --volume-id <VOLUME_ID> --size <TARGET_SIZE_GIB>

変更状況の確認。OS側の拡張は、状態が optimizing または completed になってから実施します

powershell
aws ec2 describe-volumes-modifications --volume-ids <VOLUME_ID> --query "VolumesModifications[*].[VolumeId,ModificationState,OriginalSize,TargetSize,Progress]" --output table
⚠️ 注意

EBSボリュームは拡張できますが、縮小はできません

ファイルシステムの種類とデバイス名をOS上で確認してから拡張します。誤ったデバイスに対してファイルシステム作成コマンドを実行しないでください

AWS WAF IPセットの更新

接続元IPアドレスの追加・削除が発生する場合は、Web ACLに関連付けたIPセットを更新します。変更前に現在のアドレスとロックトークンを取得し、全アドレスを含む一覧を指定して更新します。詳細は update-ip-set を確認します

値説明
<IP_SET_NAME>更新対象のAWS WAF IPセット名
<IP_SET_ID>更新対象のIPセットID
<SCOPE>REGIONAL または CLOUDFRONT
<LOCK_TOKEN>取得直後のIPセット状態を示すロックトークン

現在のアドレスとロックトークンの取得。CloudFrontスコープの場合は --region us-east-1 を追加します

powershell
aws wafv2 get-ip-set --name <IP_SET_NAME> --scope <SCOPE> --id <IP_SET_ID>

更新後アドレス一覧の作成。既存アドレスを残す場合は、必ず同じファイルに含めます

json
[
  "203.0.113.10/32",
  "198.51.100.0/24"
]

IPセットの更新

powershell
aws wafv2 update-ip-set --name <IP_SET_NAME> --scope <SCOPE> --id <IP_SET_ID> --lock-token <LOCK_TOKEN> --addresses file://ip-set-addresses.json
⚠️ 注意

update-ip-set は指定したアドレス一覧でIPセット全体を置き換えます

既存アドレスをJSONファイルから省くと、そのアドレスは削除されます。更新競合時は新しいロックトークンを取得し直して内容を再確認します

IAMユーザーの利用停止・削除

異動・退職・アカウント整理では、いきなりユーザーを削除せず、アクセスキーとコンソールログインを無効化して影響を確認します。人のログイン用途はAWS IAM Identity Centerを優先し、IAMユーザーはサービス要件がある場合に限定します

値説明
<USER_NAME>利用停止または削除するIAMユーザー名
<ACCESS_KEY_ID>無効化するアクセスキーID

アクセスキーの確認と無効化

powershell
aws iam list-access-keys --user-name <USER_NAME>
aws iam update-access-key --user-name <USER_NAME> --access-key-id <ACCESS_KEY_ID> --status Inactive

コンソールログインの停止

powershell
aws iam delete-login-profile --user-name <USER_NAME>

完全削除する場合は、アクセスキー、グループ所属、MFAデバイス、ポリシーなどの依存関係を先に解除してから delete-user を実行します
