MariaDB on RHEL 8 / 9

Red Hat Enterprise Linux 8 / 9でMariaDBを導入し、初期設定、アプリケーション用ユーザー、バックアップ、Galera Clusterの基本運用を行う手順です。

⚠️ 本番適用前の確認

MariaDBとMySQLのRPMパッケージは同一ホストへ並行導入できません。既存DB、利用中のモジュールストリーム、バックアップと復元手順を確認してから作業します。Galeraのブートストラップは、全停止からの復旧時に選定した1ノードだけで実行します。

対象と基本方針

項目内容
対象OSRed Hat Enterprise Linux 8 / 9
サービスmariadb.service
設定ファイル/etc/my.cnf と /etc/my.cnf.d/*.cnf
データディレクトリ標準では /var/lib/mysql
アプリケーション用ユーザー例t_hashimoto。rootアカウントをアプリケーションから使用しません

RHEL 8ではMariaDBをモジュールストリームとして導入します。RHEL 9では標準のmariadb-serverに加え、OSのマイナーバージョンによって追加ストリームを選択できます。導入前に利用可能なストリームを必ず確認します。

インストールと初期設定

1. OS、既存パッケージ、利用可能なストリームを確認します

shell
cat /etc/redhat-release
rpm -qa | grep -Ei '^(mariadb|mysql)'
dnf module list mariadb

2. MariaDB Serverを導入します

RHEL 8では、確認したストリーム名を指定します。RHEL 9で標準ストリームを使用する場合はmariadb-serverを導入します。既存環境では、運用中のストリームを切り替えず、サポート期限とアプリケーション互換性を確認します。

RHEL 8
sudo dnf module install mariadb:<STREAM>/server
RHEL 9 standard stream
sudo dnf install mariadb-server

3. サービスを有効化して状態を確認します

shell
sudo systemctl enable --now mariadb.service
sudo systemctl status mariadb.service --no-pager
mariadb --version
sudo ss -lntp | grep ':3306'

4. 初期セキュリティ設定を確認します

shell
sudo mariadb-secure-installation
sudo mariadb
ℹ️ root認証

MariaDB 10.4以降では、Unix socket認証が標準となる構成があります。rootのパスワードを機械的に設定するのではなく、sudo mariadbでのローカル管理可否と、匿名ユーザー・リモートrootログインの設定を確認します。

基本設定とネットワーク

個別設定は既存ファイルを直接書き換えるのではなく、用途が分かる追加ファイルへ保存します。外部接続が不要なDBは、ローカル待受を維持します。

shell
sudo cp -p /etc/my.cnf.d/mariadb-server.cnf \
  /etc/my.cnf.d/mariadb-server.cnf.bak.$(date +%Y%m%d)
sudo vi /etc/my.cnf.d/99-local.cnf
/etc/my.cnf.d/99-local.cnf
[mariadb]
bind-address = 127.0.0.1
character-set-server = utf8mb4
collation-server = utf8mb4_unicode_ci
shell
sudo systemctl restart mariadb.service
sudo journalctl -u mariadb.service -b --no-pager
sudo mariadb -e "SHOW VARIABLES WHERE Variable_name IN ('bind_address','character_set_server','collation_server');"

アプリケーションサーバーから接続する場合だけ、bind-addressを対象インターフェースのIPアドレスへ変更します。ファイアウォールは接続元CIDRを限定し、TCP 3306を全公開しません。

firewalld
sudo firewall-cmd --permanent \
  --add-rich-rule='rule family="ipv4" source address="<APP_SERVER_CIDR>" port protocol="tcp" port="3306" accept'
sudo firewall-cmd --reload
sudo firewall-cmd --list-rich-rules

データベースとユーザー管理

rootではなく、用途と接続元を限定したアカウントを作成します。以下ではアプリケーション用ユーザー名をt_hashimotoとします。パスワードはコマンド履歴に残さず、シークレット管理サービスまたは保護した設定ファイルから渡します。

SQL
CREATE DATABASE appdb
  CHARACTER SET utf8mb4
  COLLATE utf8mb4_unicode_ci;

CREATE USER 't_hashimoto'@'localhost'
  IDENTIFIED BY '<STRONG_PASSWORD>';

GRANT SELECT, INSERT, UPDATE, DELETE
  ON appdb.* TO 't_hashimoto'@'localhost';

SHOW GRANTS FOR 't_hashimoto'@'localhost';
connection test
mariadb -u t_hashimoto -p -D appdb \
  -e 'SELECT CURRENT_USER(), DATABASE();'

別ホストから接続する場合は't_hashimoto'@'<APP_SERVER_IP>'のように接続元を固定します。'%'による任意の接続元許可は使用しません。権限変更後のFLUSH PRIVILEGESは、CREATE USERやGRANTで管理した場合は不要です。

バックアップと復元

方式用途重要な確認
論理バックアップmariadb-dumpでSQLファイルを取得。単一DBの退避や検証環境への復元に向きますDBデータだけではユーザーやサーバー設定を復元できません。必要に応じてmysqlデータベース、/etc/my.cnf*も別途保管します
物理オンラインバックアップmariadb-backupで稼働中のInnoDB、Aria、MyISAMを退避します復元前に--prepareが必要です。バックアップツールとサーバーのバージョン整合性を確認します
停止中のファイルコピー保守時間を確保できる小規模環境の完全退避に使用しますサービス停止、データと設定の同時取得、同一アーキテクチャと互換バージョンでの復元が前提です

論理バックアップと復元確認

shell
sudo install -d -m 0700 /var/backups/mariadb
sudo mariadb-dump --single-transaction --routines --events --triggers \
  --databases appdb \
  > /var/backups/mariadb/appdb-$(date +%F).sql

sudo mariadb < /var/backups/mariadb/appdb-$(date +%F).sql
mariadb -u t_hashimoto -p -D appdb -e 'SHOW TABLES;'

--single-transactionはInnoDBの論理バックアップでテーブルロックを抑えるための代表的な指定です。MyISAMなど非トランザクションテーブルを含む場合は、一貫性の取り方と業務影響を別途設計します。

物理オンラインバックアップ

本番では、アプリケーション用のt_hashimotoとは分離したバックアップ専用アカウントを使用します。

SQL
CREATE USER 'mariadb_backup'@'localhost'
  IDENTIFIED BY '<STRONG_PASSWORD>';
GRANT RELOAD, LOCK TABLES, REPLICATION CLIENT
  ON *.* TO 'mariadb_backup'@'localhost';
shell
sudo dnf install mariadb-backup
sudo install -d -m 0700 /var/backups/mariadb/full
sudo mariadb-backup --backup \
  --target-dir=/var/backups/mariadb/full \
  --user=mariadb_backup --password
sudo mariadb-backup --prepare \
  --target-dir=/var/backups/mariadb/full
⚠️ 復元は別環境で先に試験します

物理復元ではMariaDBを停止し、空のデータディレクトリへ--copy-backまたは--move-backを実行します。復元後は所有者をmysql:mysqlへ戻し、SELinuxコンテキストも確認します。本番データディレクトリを直接上書きする前に、隔離環境で復元とアプリケーション接続を検証します。

MariaDB Galera Cluster

Galeraは複数ノードへ同期レプリケーションを行い、各ノードを読み書き可能にする構成です。レプリケーションはバックアップの代替ではありません。誤更新や論理削除も同期されるため、復元可能なバックアップを別に保持します。

1. 各ノードへGaleraパッケージを導入します

shell
sudo dnf install mariadb-server-galera

2. 各ノードの/etc/my.cnf.d/galera.cnfを設定します

conf
[galera]
wsrep_on=ON
wsrep_provider=/usr/lib64/galera/libgalera_smm.so
wsrep_cluster_name="mariadb_cluster"
wsrep_cluster_address="gcomm://<NODE1_PRIVATE_IP>,<NODE2_PRIVATE_IP>,<NODE3_PRIVATE_IP>"
wsrep_node_name="<NODE_NAME>"
wsrep_node_address="<NODE_PRIVATE_IP>"
binlog_format=ROW
default_storage_engine=InnoDB
innodb_autoinc_lock_mode=2
wsrep_sst_method=rsync

全ノード間でTCP 3306、TCP/UDP 4567、TCP 4568、TCP 4444を必要な範囲だけ許可します。ノードアドレス、SST方式、TLS、ロードバランサー、クォーラム喪失時の手順は構成設計として確定させます。

3. 初回ノードをブートストラップし、他ノードを参加させます

first node only
sudo galera_new_cluster
other nodes
sudo systemctl start mariadb.service
verification
sudo mariadb -e "SHOW STATUS WHERE Variable_name IN (
  'wsrep_cluster_size',
  'wsrep_cluster_status',
  'wsrep_ready',
  'wsrep_local_state_comment'
);"

正常時はwsrep_cluster_statusがPrimary、wsrep_readyがON、wsrep_local_state_commentがSyncedであることを確認します。wsrep_cluster_sizeは参加済みノード数と一致する必要があります。
