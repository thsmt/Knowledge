[構築] Red Hat Enterprise Linux 9

AWS EC2上のRed Hat Enterprise Linux 9を手動構築する際の初期設定、SSH、cloud-init、swap、ミドルウェア、Apache、Postfix、logrotateを整理します。root権限で実行する前提のため、コマンド例では sudo を省略しています

ℹ️ 前提 実環境名、IPアドレス、顧客固有値は扱わず、再利用できる手順として一般化しています。<HOSTNAME>、<FQDN>、<PRIVATE_IP> などは環境に合わせて置き換えます

1. 起動時の確認

Amazon EC2はLinuxインスタンスでRSAとED25519をサポートしています。この手順の対象環境では、組織の暗号鍵ポリシーでRSAキーペアを許可していないため、ED25519キーペアを使用します

確認項目内容
キーペアED25519を使用
接続方式SSH、Session Manager、踏み台経由など、作業経路を事前に確認します
バックアップ設定変更前にAMI、EBS Snapshot、設定ファイルのバックアップ取得要否を確認します

2. SELinuxの設定変更

Red Hat Enterprise Linux 9では SELINUX=disabled にすると起動や動作に影響する場合があるため、ログ出力のみの permissive に変更します

設定ファイルのバックアップ取得

root shell
mkdir -p /root/config-backup/selinux
cp -pi /etc/selinux/config /root/config-backup/selinux/config.bak

設定ファイルの編集

root shell
vi /etc/selinux/config

SELINUXのpermissiveへの変更

conf
SELINUX=permissive

変更差分の確認

root shell
diff -u /root/config-backup/selinux/config.bak /etc/selinux/config

再起動前の一時反映

root shell
setenforce permissive

動作状態の確認

root shell
getenforce

3. subscription-managerの無効化

AWS Marketplaceの従量課金（PAYG）RHEL AMIは、Red Hat Update Infrastructure（RHUI）から更新パッケージを取得し、Red Hat Subscription Management（RHSM）への登録を必要としません。この構成ではsubscription-managerプラグインを無効化します

設定ファイルのバックアップ取得

root shell
mkdir -p /root/config-backup/yum-plugin
cp -pi /etc/yum/pluginconf.d/subscription-manager.conf /root/config-backup/yum-plugin/subscription-manager.conf.bak

設定ファイルの編集

root shell
vi /etc/yum/pluginconf.d/subscription-manager.conf

enabledの1から0への変更

conf
[main]
enabled=0
ℹ️ 情報

PAYG RHEL AMIで有効のままにすると、This system is not registered with an entitlement server.という未登録警告がDNF実行時に表示されます。RHUIからの更新は継続できますが、作業ログへ不要な警告が混在し、実際のリポジトリエラーを判別しづらくなります

BYOS、RHEL Gold Image、Red Hat Satellite、Red Hat CDNへ登録してサブスクリプションを管理する環境では、登録状態とリポジトリ制御に使用するため無効化しません

4. ログインプロファイル設定

操作履歴に時刻を残し、調査時に作業順序を追いやすくするため、/etc/profile と新規ユーザー用の雛形を変更します

/etc/profileのバックアップ取得

/etc/profileの編集

履歴件数、履歴時刻、umaskの設定

/etc/profile の設定例を展開する
conf
# /etc/profile
pathmunge () {
    case ":${PATH}:" in
        *:"$1":*)
            ;;
        *)
            if [ "$2" = "after" ] ; then
                PATH=$PATH:$1
            else
                PATH=$1:$PATH
            fi
    esac
}

if [ -x /usr/bin/id ]; then
    if [ -z "$EUID" ]; then
        EUID=`id -u`
        UID=`id -ru`
    fi
    USER="`id -un`"
    LOGNAME=$USER
    MAIL="/var/spool/mail/$USER"
fi

if [ "$EUID" = "0" ]; then
    pathmunge /sbin
    pathmunge /usr/sbin
    pathmunge /usr/local/sbin
else
    pathmunge /usr/local/sbin after
    pathmunge /usr/sbin after
    pathmunge /sbin after
fi

HOSTNAME=`/bin/hostname 2>/dev/null`
HISTSIZE=50000
HISTFILESIZE=50000
if [ "$HISTCONTROL" = "ignorespace" ] ; then
    export HISTCONTROL=ignoreboth
else
    export HISTCONTROL=ignoredups
fi
export PATH USER LOGNAME MAIL HOSTNAME HISTSIZE HISTFILESIZE HISTCONTROL

if [ $UID -gt 199 ] && [ "`id -gn`" = "`id -un`" ]; then
    umask 002
else
    umask 022
fi

for i in /etc/profile.d/*.sh ; do
    if [ -r "$i" ]; then
        if [ "${-#*i}" != "$-" ]; then
            . "$i"
        else
            . "$i" >/dev/null 2>&1
        fi
    fi
done

unset i
unset -f pathmunge
umask 002
HISTTIMEFORMAT='%Y-%m-%d %T '

8. cloud-init設定の確認・必要時変更

cloud-initはEC2の初回起動時に、インスタンスメタデータとユーザーデータを基に、既定ユーザー、SSH鍵、ホスト名、パッケージ導入などを反映します。既定動作で要件を満たす場合は変更不要です

⚠️ 注意

cloud.cfgの変更は、以後の初回起動処理やAMIから複製したインスタンスの動作に影響します

既定ユーザー、rootログイン、SSH鍵、ユーザーデータの実行順序を要件として変更する場合だけ、変更前後の差分を確認して実施します

9. 追加ディスクのマウント

EBSボリュームを追加する場合は、デバイス確認、ファイルシステム作成、マウントポイント作成、/etc/fstab追記まで実施します。NitroベースのEC2では、コンソール上のデバイス名とOS上のデバイス名が異なるため、必ずlsblkで対象を特定します

追加ディスクの確認

root shell
lsblk -f
file -s /dev/<DEVICE_NAME>

ファイルシステムとマウントポイントの作成

root shell
mkfs -t xfs /dev/<DEVICE_NAME>
mkdir -p /data

一時マウントとUUID確認

root shell
mount /dev/<DEVICE_NAME> /data
blkid /dev/<DEVICE_NAME>

/etc/fstabへの永続化設定

fstab
UUID=<UUID> /data xfs defaults,nofail 0 2

マウント設定の検証

root shell
mount -a
df -hT /data
⚠️ 注意 既存のデータがあるデバイスにmkfsを実行するとデータを失います。file -sとlsblk -fの結果が空の新規ボリュームであることを確認してから実行します

Amazon EBSボリュームの使用

ℹ️ 情報

prefork MPMは接続ごとにプロセスを割り当てるため、同時接続数が増えるほどメモリ使用量が増えます

非スレッドセーフなモジュールや従来のmod_php構成が必要な場合を除き、Red Hat Enterprise Linux 9ではPHP-FPMと組み合わせやすいevent MPMを優先します。preforkは各プロセスが1リクエストを処理するため、同じ同時接続数ではスレッド型のevent MPMより多くのメモリを消費します

preforkが必要な場合は、MaxRequestWorkersと1プロセス当たりのメモリ使用量から上限を算出し、対象モジュールと負荷条件を事前に検証します

自動起動の確認と有効化

root shell
systemctl is-enabled httpd.service
systemctl enable httpd.service
systemctl is-enabled httpd.service

14. 運用支援ツールの導入

運用・障害対応で利用する補助ツールの導入。Red Hat Enterprise Linux 9ではyumもDNFへの互換コマンドですが、この手順ではdnfを使用します

root shell
dnf install -y telnet lftp mlocate libyaml wget mtr tcpdump tmux bash-completion net-tools

net-tools旧来のnetstat・ifconfig。通常はss・ipを優先します

15. logrotate設定

既存設定のバックアップ取得

root shell
mkdir -p /root/config-backup/logrotate
cp -pi /etc/logrotate.conf /root/config-backup/logrotate/logrotate.conf.bak

logrotate設定の編集

root shell
vi /etc/logrotate.conf

/etc/logrotate.confの設定反映

conf
# see "man logrotate" for details
weekly
rotate 54
create
dateext
compress
include /etc/logrotate.d
# system-specific logs may be also be configured here.
⚠️ 注意

/etc/logrotate.confは/etc/logrotate.dを読み込みます。たとえば/etc/logrotate.d/httpdを同じディレクトリへhttpd.bakとして保存すると、設定内容が二重に読み込まれることがあります

同じログに対するローテーションやpostrotateが重複し、エラーや意図しないサービス再読み込みにつながるため、バックアップは/root/config-backup/logrotate/のように別ディレクトリへ保存します
