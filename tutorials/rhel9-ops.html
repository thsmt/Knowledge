RHEL9 運用手順

AWS EC2上のRHEL9を手動構築・保守する際の確認観点を整理します。実環境名、IPアドレス、顧客固有値は扱わず、再利用できる作業パターンだけをまとめます。

方針 手動手順はAnsible化する前の確認用として扱います。最終的には、パッケージ導入、設定ファイル変更、サービス再起動、ログ確認をPlaybookへ移行できる粒度で整理します。

作業前確認

観点確認内容
対象確認アカウント、リージョン、インスタンスID、OSバージョン、ホスト名を確認します。
退避AMI、EBS Snapshot、設定ファイルのバックアップ要否を確認します。
接続経路Session Manager、踏み台、Security Group一時許可のどれを使うか確認します。
影響サービス再起動、パッケージ更新、カーネル更新の有無を確認します。

基本パッケージと確認コマンド

bash
cat /etc/redhat-release
hostnamectl
ip addr
df -h
free -m
sudo dnf check-update
sudo dnf install -y telnet lftp mlocate wget mtr tcpdump tmux bash-completion net-tools

logrotate設定

ログ保持期間や圧縮有無は環境ごとに差が出ます。変更前に既存設定を退避し、変更後に構文とローテーション対象を確認します。

bash
sudo cp -pi /etc/logrotate.conf /etc/logrotate.conf.$(date +%Y%m%d)
sudo vi /etc/logrotate.conf
sudo logrotate -d /etc/logrotate.conf

SSM Agent確認

Session ManagerやPatch Managerを使う場合は、SSM Agent、IAM Role、VPC Endpointまたはインターネット経路を確認します。

bash
systemctl status amazon-ssm-agent
sudo systemctl enable amazon-ssm-agent
sudo systemctl restart amazon-ssm-agent

CloudWatch Agent更新

Agent更新では、更新前バージョン、設定ファイル、更新後のサービス状態、CloudWatch Logsへの到達を確認します。

bash
rpm -qa | grep amazon-cloudwatch-agent
sudo dnf update -y amazon-cloudwatch-agent
sudo systemctl status amazon-cloudwatch-agent

Apache更新・設定変更

Apacheの脆弱性対応や設定変更では、更新前後のRPM、構文確認、プロセス確認、ステータス確認を残します。

bash
rpm -qa | grep httpd
sudo apachectl configtest
sudo systemctl status httpd
sudo dnf update -y httpd
sudo apachectl configtest
sudo systemctl restart httpd
sudo systemctl status httpd

作業前後エビデンス

作業前後で同じ観点を比較します。コマンド結果、AWSコンソールの状態、サービス応答、ログ出力を同じ順序で残すとレビューしやすくなります。
