AWS CLI 実務手順集

AWS CLIを使った認証確認、リージョン設定、IAM、Security Group、SNS、Elastic IP、Compute Optimizerの基本手順を整理します。

補足 AWS CLIの作業前には、利用者、アカウントID、リージョンを確認します。CloudShellや作業端末にデフォルト認証情報が設定済みであれば、以降のコマンドに --profile は不要です。

AWS CLIの初期設定

インストール

作業端末のOSに合わせてAWS CLIをインストールします。

powershell
winget install --id Amazon.AWSCLI -e --source winget
aws --version
zsh
brew install awscli
aws --version
bash
curl "https://awscli.amazonaws.com/awscli-exe-linux-x86_64.zip" -o "awscliv2.zip"
unzip awscliv2.zip
sudo ./aws/install
aws --version

認証情報の確認

現在の認証情報が、想定しているAWSアカウントとIAMユーザー/ロールを向いているか確認します。

powershell
aws sts get-caller-identity

認証情報の設定

作業端末で初めてAWS CLIを使う場合は、デフォルト認証情報を設定します。

powershell
aws configure
補足 --profile は、複数のAWSアカウントやIAMユーザーを名前付きプロファイルで切り替える場合だけ指定します。このページでは、デフォルト認証情報を使う前提で記載します。

リージョン確認

作業前に対象リージョンを明示します。東京リージョンで作業する場合は ap-northeast-1 を指定します。

powershell
aws configure set region ap-northeast-1

AWS Identity and Access Management (IAM) / AWS IAM Identity Center

原則として人のログイン用途はAWS IAM Identity Centerを優先します。AWS Identity and Access Management (IAM) のIAMユーザーを作成する場合は、用途、権限、利用期限、アクセスキーの有無を事前に確認します。

値説明
<USER_NAME>作成または確認するIAMユーザー名
<GROUP_NAME>ユーザーを所属させるIAMグループ名

IAMユーザーを作成します。

powershell
aws iam create-user --user-name <USER_NAME>

作成したIAMユーザーを既存グループに追加します。

powershell
aws iam add-user-to-group --user-name <USER_NAME> --group-name <GROUP_NAME>

ユーザーの所属グループを確認します。

powershell
aws iam list-groups-for-user --user-name <USER_NAME>

Security Group ルール追加・削除

Security Group変更は影響範囲が広いため、対象SG、方向、プロトコル、ポート、送信元/宛先、説明欄を事前に確定します。

AWS CLIのSecurity Groupルールで全プロトコルを指定する場合は、公式リファレンスに合わせて --protocol -1 を使用します。

値説明
<SECURITY_GROUP_ID>変更対象のSecurity Group ID
<CIDR>許可または削除する送信元CIDR。例: 203.0.113.10/32

作業前に、対象Security Groupの現在のルールを確認します。

powershell
aws ec2 describe-security-groups --group-ids <SECURITY_GROUP_ID>

HTTPの80/tcpだけを許可する例です。

powershell
aws ec2 authorize-security-group-ingress --group-id <SECURITY_GROUP_ID> --protocol tcp --port 80 --cidr <CIDR>

全プロトコル・全ポートを許可する例です。検証や一時作業以外では原則として避け、作業後に必ず削除します。

powershell
aws ec2 authorize-security-group-ingress --group-id <SECURITY_GROUP_ID> --protocol -1 --cidr <CIDR>

80/tcpの許可ルールを削除する例です。

powershell
aws ec2 revoke-security-group-ingress --group-id <SECURITY_GROUP_ID> --protocol tcp --port 80 --cidr <CIDR>

全プロトコル・全ポートの許可ルールを削除する例です。

powershell
aws ec2 revoke-security-group-ingress --group-id <SECURITY_GROUP_ID> --protocol -1 --cidr <CIDR>

Prefix List 確認

Security GroupやRoute TableでPrefix Listを参照している場合は、Prefix List ID、名前、登録CIDRを確認します。詳細は describe-managed-prefix-lists と get-managed-prefix-list-entries を確認します。

値説明
<PREFIX_LIST_NAME>確認対象のPrefix List名
<PREFIX_LIST_ID>確認対象のPrefix List ID

利用可能なPrefix Listの一覧を確認します。

powershell
aws ec2 describe-managed-prefix-lists --query "PrefixLists[*].[PrefixListId,PrefixListName,AddressFamily,State,OwnerId]" --output table

Prefix List名で絞り込んで確認します。

powershell
aws ec2 describe-managed-prefix-lists --filters Name=prefix-list-name,Values=<PREFIX_LIST_NAME> --query "PrefixLists[*].[PrefixListId,PrefixListName,State]" --output table

Prefix Listに登録されているCIDRを確認します。

powershell
aws ec2 get-managed-prefix-list-entries --prefix-list-id <PREFIX_LIST_ID> --query "Entries[*].[Cidr,Description]" --output table

SNS Topic・Subscription作成

通知先を作成する場合は、Topic、Subscription、確認メールの承認、通知テストまでを一連の手順として扱います。

SNS Topicを作成します。

powershell
aws sns create-topic --name <TOPIC_NAME>

メール通知先をSubscriptionとして登録します。

powershell
aws sns subscribe --topic-arn <TOPIC_ARN> --protocol email --notification-endpoint <MAIL_ADDRESS>

通知テストを実行します。

powershell
aws sns publish --topic-arn <TOPIC_ARN> --message "notification test"

RDS 保留中の変更確認・取り消し

RDSのエンジンバージョン変更を「次回のメンテナンスウィンドウで適用」として保存した後、コンソール上で保留中の変更を取り消せない場合があります。その場合は、AWS CLIで現在稼働中の値を再指定して、保留中の変更を打ち消します。詳細は describe-db-instances と modify-db-instance を確認します。

値説明
<DB_INSTANCE_IDENTIFIER>対象のRDS DBインスタンス識別子
<CURRENT_ENGINE_VERSION>現在稼働中のDBエンジンバージョン。例: 13.11

対象DBインスタンスの現在のエンジンバージョンと、保留中の変更を確認します。

powershell
aws rds describe-db-instances --db-instance-identifier <DB_INSTANCE_IDENTIFIER> --query "DBInstances[*].[DBInstanceIdentifier,EngineVersion,PendingModifiedValues]" --output table

保留中のエンジンバージョン変更を取り消すため、現在稼働中のエンジンバージョンを再指定します。

powershell
aws rds modify-db-instance --db-instance-identifier <DB_INSTANCE_IDENTIFIER> --engine-version <CURRENT_ENGINE_VERSION> --apply-immediately

変更後に、PendingModifiedValuesが空になっていることを確認します。

powershell
aws rds describe-db-instances --db-instance-identifier <DB_INSTANCE_IDENTIFIER> --query "DBInstances[*].[DBInstanceIdentifier,EngineVersion,PendingModifiedValues]" --output table
注意 エンジンバージョン変更は停止を伴う場合があります。作業前にメンテナンスウィンドウ、バックアップ、スナップショット、対象バージョン、保留中の変更内容を確認します。エンジンバージョン以外の保留中変更は、対象パラメータごとに現在値を再指定します。

Elastic IP 解放後の再取得

Elastic IPを誤って解放した場合でも、他のAWSアカウントに割り当てられていなければ、AWS CLIで同じPublic IPv4アドレスを指定して再取得できる場合があります。詳細は Amazon EC2 User Guide を確認します。

値説明
<PUBLIC_IP>再取得したいElastic IPアドレス
<REGION>Elastic IPを利用していたリージョン
<ALLOCATION_ID>再取得後に払い出されたElastic IPのAllocation ID
<INSTANCE_ID>Elastic IPを関連付けるEC2インスタンスID

解放済みのElastic IPを同じアドレスで再取得します。

powershell
aws ec2 allocate-address --domain vpc --address <PUBLIC_IP> --region <REGION>

再取得したElastic IPをEC2インスタンスへ関連付ける場合は、払い出されたAllocation IDを指定します。

powershell
aws ec2 associate-address --allocation-id <ALLOCATION_ID> --instance-id <INSTANCE_ID> --region <REGION>
注意 解放済みのElastic IPが他のAWSアカウントに割り当てられている場合は再取得できません。作業前に対象リージョン、対象IP、現在の割り当て状況を確認します。

Compute Optimizer オプトアウト

Compute Optimizerからアカウントをオプトアウトする場合は、AWS CLIでEnrollment statusをInactiveに更新します。詳細は AWS Compute Optimizer User Guide を確認します。

現在のCompute Optimizer登録状態を確認します。

powershell
aws compute-optimizer get-enrollment-status

Compute Optimizerからアカウントをオプトアウトします。

powershell
aws compute-optimizer update-enrollment-status --status Inactive
注意 オプトアウト時に --include-member-accounts を指定するとエラーになります。また、オプトアウトすると対象アカウントのレコメンデーションと関連メトリクスデータはCompute Optimizerから削除されます。
