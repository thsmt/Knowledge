AWS CLI 特殊対応

AWS CLI/APIでしか変更・切り戻し・復旧できない運用手順を整理します

📍 掲載方針

このページでは、CLI/APIでしか変更または切り戻しできない操作を扱います。Security GroupやSNSなどの日常的な基本コマンドは、AWS CLI コマンドリファレンス に分けています

Amazon RDS 保留中の変更の取り消し

Amazon RDSのエンジンバージョン変更を「次回のメンテナンスウィンドウで適用」として保存した後、コンソール上で保留中の変更を取り消せない場合があります。その場合は、AWS CLIで現在稼働中の値を再指定して、保留中の変更を打ち消します。詳細は describe-db-instances と modify-db-instance を確認します

値説明
<DB_INSTANCE_IDENTIFIER>対象のRDS DBインスタンス識別子
<CURRENT_ENGINE_VERSION>現在稼働中のDBエンジンバージョン。例: 13.11

対象DBインスタンスの現在のエンジンバージョンと、保留中の変更を確認します

powershell
aws rds describe-db-instances --db-instance-identifier <DB_INSTANCE_IDENTIFIER> --query "DBInstances[0].{DBInstanceIdentifier:DBInstanceIdentifier,EngineVersion:EngineVersion,PendingModifiedValues:PendingModifiedValues}" --output json

保留中のエンジンバージョン変更を取り消すため、現在稼働中のエンジンバージョンを再指定します

powershell
aws rds modify-db-instance --db-instance-identifier <DB_INSTANCE_IDENTIFIER> --engine-version <CURRENT_ENGINE_VERSION> --apply-immediately

変更後に、PendingModifiedValues が {} になっていることを確認します

powershell
aws rds describe-db-instances --db-instance-identifier <DB_INSTANCE_IDENTIFIER> --query "DBInstances[0].{DBInstanceIdentifier:DBInstanceIdentifier,EngineVersion:EngineVersion,PendingModifiedValues:PendingModifiedValues}" --output json
実行例を展開する

検証では、Amazon RDS for PostgreSQLの保留中のエンジンバージョン変更を、現在稼働中の 17.8 を再指定して取り消しました

powershell
aws rds describe-db-instances --db-instance-identifier database-1 --query "DBInstances[0].{DBInstanceIdentifier:DBInstanceIdentifier,EngineVersion:EngineVersion,PendingModifiedValues:PendingModifiedValues}" --output json
{
  "DBInstanceIdentifier": "database-1",
  "EngineVersion": "17.8",
  "PendingModifiedValues": {
    "EngineVersion": "17.9"
  }
}

aws rds modify-db-instance --db-instance-identifier database-1 --engine-version 17.8 --apply-immediately

aws rds describe-db-instances --db-instance-identifier database-1 --query "DBInstances[0].{DBInstanceIdentifier:DBInstanceIdentifier,EngineVersion:EngineVersion,PendingModifiedValues:PendingModifiedValues}" --output json
{
  "DBInstanceIdentifier": "database-1",
  "EngineVersion": "17.8",
  "PendingModifiedValues": {}
}
⚠️ 注意

エンジンバージョン変更は停止を伴う場合があります

--apply-immediately を指定した場合は今回再指定した項目だけでなく、同一DBインスタンスに残っている他の保留中変更も即時適用される場合があります

作業前に PendingModifiedValues 全体を確認し、対象外の保留中変更がある場合は作業を中止するか、取り消す各項目の現在値を同一リクエストで再指定します

EC2 EBS終了時削除属性の変更

EC2起動後にEBSボリュームの DeleteOnTermination 属性を変更する場合は、AWS CLIで modify-instance-attribute を実行します。追加EBSボリュームは終了時削除が無効になっている場合があるため、検証環境や一時サーバーでは削除漏れ防止のために確認します。詳細は ModifyInstanceAttribute のAWS公式例 と modify-instance-attribute を確認します

値説明
<INSTANCE_ID>設定を変更するEC2インスタンスID
<DEVICE_NAME>対象EBSボリュームのDeviceName。例: /dev/sdf、/dev/xvdf

現在のBlockDeviceMappingと、終了時削除属性を確認します

powershell
aws ec2 describe-instances --instance-ids <INSTANCE_ID> --query "Reservations[0].Instances[0].BlockDeviceMappings[*].{DeviceName:DeviceName,VolumeId:Ebs.VolumeId,DeleteOnTermination:Ebs.DeleteOnTermination}" --output table

対象EBSボリュームを、EC2終了時に削除される設定へ変更します

powershell
aws ec2 modify-instance-attribute --instance-id <INSTANCE_ID> --block-device-mappings '[{"DeviceName":"<DEVICE_NAME>","Ebs":{"DeleteOnTermination":true}}]'

データ保持が必要なボリュームの場合は、終了時に削除されない設定へ戻します

powershell
aws ec2 modify-instance-attribute --instance-id <INSTANCE_ID> --block-device-mappings '[{"DeviceName":"<DEVICE_NAME>","Ebs":{"DeleteOnTermination":false}}]'

変更後に、対象DeviceNameの DeleteOnTermination が想定した値になっていることを確認します

powershell
aws ec2 describe-instances --instance-ids <INSTANCE_ID> --query "Reservations[0].Instances[0].BlockDeviceMappings[*].{DeviceName:DeviceName,VolumeId:Ebs.VolumeId,DeleteOnTermination:Ebs.DeleteOnTermination}" --output table
⚠️ 注意 modify-instance-attribute では、Volume IDではなくインスタンス側の DeviceName を指定します。終了時削除を有効にすると、EC2インスタンスをterminateしたタイミングで対象EBSボリュームも削除されます。DB、ログ、永続データを保持するボリュームでは有効化しないよう、作業前に用途を確認します

Elastic IP 解放後の再取得

Elastic IPを誤って解放した場合でも、他のAWSアカウントに割り当てられていなければ、AWS CLIで同じPublic IPv4アドレスを指定して再取得できる場合があります。詳細は Amazon EC2 User Guide を確認します

値説明
<PUBLIC_IP>再取得したいElastic IPアドレス
<REGION>Elastic IPを利用していたリージョン
<ALLOCATION_ID>再取得後に払い出されたElastic IPのAllocation ID
<INSTANCE_ID>Elastic IPを関連付けるEC2インスタンスID

解放済みのElastic IPを同じアドレスで再取得します

powershell
aws ec2 allocate-address --domain vpc --address <PUBLIC_IP> --region <REGION>

再取得したElastic IPをEC2インスタンスへ関連付ける場合は、払い出されたAllocation IDを指定します

powershell
aws ec2 associate-address --allocation-id <ALLOCATION_ID> --instance-id <INSTANCE_ID> --region <REGION>
⚠️ 注意 解放済みのElastic IPが他のAWSアカウントに割り当てられている場合は再取得できません。作業前に対象リージョン、対象IP、現在の割り当て状況を確認します

Amazon S3 Versioning環境で削除されたフォルダを復元

S3 Versioningが有効なバケットでは、オブジェクト削除時にDelete Markerが作成されます。S3に実体としてのフォルダはないため、フォルダ復元は指定Prefix配下のDelete Markerを確認し、対象のDelete MarkerをVersion ID付きで削除する作業として扱います。詳細は Restoring previous versions と list-object-versions を確認します

値説明
<BUCKET_NAME>Versioningが有効なS3バケット名
<PREFIX>復元対象のPrefix。例: logs/app
<OBJECT_KEY>復元対象オブジェクトのKey
<DELETE_MARKER_VERSION_ID>削除するDelete MarkerのVersion ID

作業前に、対象バケットのVersioning状態を確認します

powershell
aws s3api get-bucket-versioning --bucket <BUCKET_NAME>

指定Prefix配下で、最新バージョンがDelete Markerになっているオブジェクトを確認します

powershell
aws s3api list-object-versions --bucket <BUCKET_NAME> --prefix "<PREFIX>/" --query 'DeleteMarkers[?IsLatest==`true`].[Key,VersionId,LastModified]' --output table

対象オブジェクトのDelete MarkerをVersion ID付きで削除し、直前のオブジェクトバージョンを見える状態に戻します

powershell
aws s3api delete-object --bucket <BUCKET_NAME> --key "<OBJECT_KEY>" --version-id <DELETE_MARKER_VERSION_ID>

復元後に、対象オブジェクトが参照できることを確認します

powershell
aws s3api head-object --bucket <BUCKET_NAME> --key "<OBJECT_KEY>"
⚠️ 注意 --version-id を付けずに delete-object を実行すると、Delete Markerを削除するのではなく、新しいDelete Markerを追加する動作になります。復元作業では、削除対象がDelete Marker自身のVersion IDであることを必ず確認します

IAM仮想MFAデバイスの未割当て状態を解消

IAMコンソールで仮想MFAデバイス登録を途中で中断すると、IAMユーザーに割り当てられていない仮想MFAデバイスが残る場合があります。同じ名前で再登録できない場合は、AWS CLIで未割当てデバイスを確認して削除します。詳細は list-virtual-mfa-devices と delete-virtual-mfa-device を確認します

値説明
<VIRTUAL_MFA_DEVICE_ARN>削除対象の仮想MFAデバイスARN

ユーザーに割り当てられていない仮想MFAデバイスを確認します

powershell
aws iam list-virtual-mfa-devices --assignment-status Unassigned --query "VirtualMFADevices[*].SerialNumber" --output table

不要な未割当て仮想MFAデバイスを削除します

powershell
aws iam delete-virtual-mfa-device --serial-number <VIRTUAL_MFA_DEVICE_ARN>

削除後に、未割当て仮想MFAデバイス一覧から消えていることを確認します

powershell
aws iam list-virtual-mfa-devices --assignment-status Unassigned --query "VirtualMFADevices[*].SerialNumber" --output table
⚠️ 注意 IAMユーザーに割り当て済みのMFAデバイスは、先に割り当て解除が必要です。この手順は、登録途中で残った未割当てMFAデバイスの整理を目的とします

Auto Scalingライフサイクルアクションの待機時間を延長・完了

Auto Scalingのライフサイクルフックでインスタンスが Pending:Wait または Terminating:Wait になっている場合、作業時間の延長や処理完了通知をAWS CLIで実行します。デプロイ前の初期設定、監視エージェント登録、終了前のログ退避などで利用します。詳細は record-lifecycle-action-heartbeat と complete-lifecycle-action を確認します

値説明
<AUTO_SCALING_GROUP_NAME>対象のAuto Scaling Group名
<LIFECYCLE_HOOK_NAME>対象のLifecycle Hook名
<INSTANCE_ID>待機中のEC2インスタンスID

対象Auto Scaling GroupのLifecycle Hookを確認します

powershell
aws autoscaling describe-lifecycle-hooks --auto-scaling-group-name <AUTO_SCALING_GROUP_NAME>

作業時間が不足する場合は、ライフサイクルアクションのハートビートを送信して待機時間を延長します

powershell
aws autoscaling record-lifecycle-action-heartbeat --lifecycle-hook-name <LIFECYCLE_HOOK_NAME> --auto-scaling-group-name <AUTO_SCALING_GROUP_NAME> --instance-id <INSTANCE_ID>

作業完了後、インスタンスの起動または終了処理を続行します

powershell
aws autoscaling complete-lifecycle-action --lifecycle-action-result CONTINUE --lifecycle-hook-name <LIFECYCLE_HOOK_NAME> --auto-scaling-group-name <AUTO_SCALING_GROUP_NAME> --instance-id <INSTANCE_ID>

作業を中止する場合は、設計に応じて ABANDON を指定します

powershell
aws autoscaling complete-lifecycle-action --lifecycle-action-result ABANDON --lifecycle-hook-name <LIFECYCLE_HOOK_NAME> --auto-scaling-group-name <AUTO_SCALING_GROUP_NAME> --instance-id <INSTANCE_ID>
⚠️ 注意 CONTINUE と ABANDON の意味は、起動時と終了時で運用影響が異なります。事前にLifecycle Hookの目的、通知先、タイムアウト、デフォルト結果を確認してから実行します

既存CloudWatch LogsロググループへKMSキーを関連付け

既存のCloudWatch LogsロググループにCustomer Managed KMS Keyを関連付ける場合は、AWS CLIで associate-kms-key を実行します。監査要件や暗号化キー管理を後から整備する場合に利用します。詳細は Encrypt log data in CloudWatch Logs using AWS KMS と associate-kms-key を確認します

値説明
<LOG_GROUP_NAME>KMSキーを関連付けるCloudWatch Logsロググループ名
<KMS_KEY_ARN>CloudWatch Logsで利用する対称KMSキーのARN

作業前に、ロググループへKMSキーが関連付け済みか確認します

powershell
aws logs describe-log-groups --log-group-name-prefix "<LOG_GROUP_NAME>" --query "logGroups[*].{LogGroupName:logGroupName,KmsKeyId:kmsKeyId}" --output table

既存ロググループへKMSキーを関連付けます

powershell
aws logs associate-kms-key --log-group-name "<LOG_GROUP_NAME>" --kms-key-id "<KMS_KEY_ARN>"

関連付けを解除する場合は、ロググループ名を指定して解除します

powershell
aws logs disassociate-kms-key --log-group-name "<LOG_GROUP_NAME>"
⚠️ 注意 CloudWatch Logsで使用できるのは対称KMSキーです。反映に数分かかる場合があり、関連付け後に取り込まれたログデータが対象KMSキーで暗号化されます。KMSキーを無効化または削除すると、そのキーで暗号化されたログを読み出せなくなるため、キーのライフサイクル管理も確認します

既存CloudFormationスタックへスタックポリシーを設定

CloudFormation Stack Policyを使うと、スタック更新時に特定のLogical IDを誤って変更しないよう保護できます。既存スタックへStack Policyを追加または置き換える場合は、AWS CLIで設定します。詳細は Prevent updates to stack resources と set-stack-policy を確認します

値説明
<STACK_NAME>Stack Policyを設定するCloudFormationスタック名
<LOGICAL_RESOURCE_ID>更新から保護するリソースのLogical ID

保護対象のLogical IDを確認します

powershell
aws cloudformation describe-stack-resources --stack-name <STACK_NAME> --query "StackResources[*].[LogicalResourceId,ResourceType,PhysicalResourceId]" --output table

特定リソースの更新を拒否するStack Policyを作成します

json
{
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "Update:*",
      "Principal": "*",
      "Resource": "*"
    },
    {
      "Effect": "Deny",
      "Action": "Update:*",
      "Principal": "*",
      "Resource": "LogicalResourceId/<LOGICAL_RESOURCE_ID>"
    }
  ]
}

既存スタックへStack Policyを設定します

powershell
aws cloudformation set-stack-policy --stack-name <STACK_NAME> --stack-policy-body file://stack-policy.json

設定済みのStack Policyを確認します

powershell
aws cloudformation get-stack-policy --stack-name <STACK_NAME>
⚠️ 注意 Stack Policyは削除ではなく置き換えで管理します。保護を解除する場合も、すべての更新を許可するPolicyへ置き換える運用になります。設定前に、対象Logical IDと更新計画を必ず確認します

Compute Optimizer オプトアウト

Compute Optimizerからアカウントをオプトアウトする場合は、AWS CLIでEnrollment statusをInactiveに更新します。詳細は AWS Compute Optimizer User Guide を確認します

現在のCompute Optimizer登録状態を確認します

powershell
aws compute-optimizer get-enrollment-status

Compute Optimizerからアカウントをオプトアウトします

powershell
aws compute-optimizer update-enrollment-status --status Inactive
⚠️ 注意 オプトアウト時に --include-member-accounts を指定するとエラーになります。また、オプトアウトすると対象アカウントのレコメンデーションと関連メトリクスデータはCompute Optimizerから削除されます
