AWS Certificate Manager (ACM) と証明書運用

AWS Certificate Managerで扱うパブリック証明書、エクスポート可能な証明書、ACME証明書、インポート証明書の違いを整理します。証明書の発行だけでなく、更新、サーバーへの再配置、監視、失効時の対応までを運用対象とします。

⚠️ 2026年7月時点の重要な変更

ACMパブリック証明書の有効期間は、2026年2月18日以降、従来の395日から198日へ短縮されています。ACMが更新を開始する時期も期限60日前から45日前へ変わりました。2025年の記事にある「395日」「1 FQDNあたり15 USD」「ワイルドカード149 USD」は現在の仕様・価格ではありません。

ℹ️ サーバーに配置した証明書は自動で置き換わりません

ACMがエクスポート可能な証明書を更新しても、EC2上のApacheやNGINXに置いた証明書ファイルは更新されません。更新後の証明書を再エクスポートし、対象サーバーへ配置して、構文確認とサービス再読み込みを行う仕組みが必要です。

ACMで扱う証明書

種類主な用途更新と配置
ACMパブリック証明書ALB、CloudFront、API GatewayなどのACM統合サービスACMが更新し、統合サービスへ自動配置します
エクスポート可能なパブリック証明書EC2、コンテナ、オンプレミスなど、利用者がTLSを終端する環境ACMは更新しますが、再エクスポートと再配置は利用者が行います
ACME証明書Certbot、cert-manager、acme.shを使う利用者管理環境ACMEクライアントが発行、更新、サーバー配置を管理します
インポート証明書外部認証局が発行した証明書をAWSサービスで利用する場合ACMは更新しません。期限監視、再発行、再インポートが必要です
AWS Private CAの証明書社内システム、相互TLS、内部サービス発行経路と利用先により更新方法が異なります。クライアント側へのCA証明書配布も必要です

同じドメイン名でも、証明書はリージョンごとのリソースです。CloudFrontで使用する証明書はバージニア北部リージョン（us-east-1）で管理します。

有効期間と今後の短縮

時期公開TLS証明書の最大有効期間運用への影響
2026年2月18日以降ACMパブリック証明書は198日ACMは期限45日前から更新を試行します
2027年3月15日以降100日未満手動更新では作業頻度と失効リスクが大きくなります
2029年3月15日以降47日未満発行、配置、確認、失敗通知までの自動化が前提になります

既存の395日または398日証明書は、更新または失効まで引き続き利用できます。更新後は198日証明書へ切り替わります。

ACM ACMEで発行する証明書は、将来の短期化を見据えて既に45日間です。短期証明書では、更新ジョブの成功だけでなく、外部接続時に返される証明書の有効期限も監視します。

方式の選び方

構成第一候補理由
ALB、CloudFront、API Gateway通常のACMパブリック証明書秘密鍵を取り出さず、更新と配置をAWSへ任せられます
EC2上のApache / NGINXACM ACMEACMEクライアントがサーバー上で発行と更新を自動化できます
既存のACM管理方式をサーバーへ拡張エクスポート可能な証明書とAWS Workload Credentials Provider定期的な再エクスポート、ファイル更新、サービス再読み込みを自動化できます
複数サーバーへ独自の配布制御が必要EventBridge、Lambda、Systems Manager更新イベントを起点に、タグや台帳を使って配布先を制御できます
第三者認証局の証明書インポートと独自更新手順ACMのマネージド更新対象ではありません

EC2でTLSを直接終端する必要がなければ、ALBでTLSを終端して通常のACM証明書を関連付ける構成が最も管理項目を減らせます。

AWS統合サービスでの自動更新

ACM統合サービスに関連付けたAmazon発行証明書は、更新条件を満たす場合、ACMが更新し、統合サービスへ新しい証明書を配置します。DNS検証を使用する場合は、検証用CNAMEレコードを削除しません。

AWS CLI
aws acm describe-certificate \
  --certificate-arn <CERTIFICATE_ARN> \
  --region ap-northeast-1 \
  --query 'Certificate.{Status:Status,InUseBy:InUseBy,NotAfter:NotAfter,RenewalStatus:RenewalSummary.RenewalStatus}'

ACMが更新可能でも、削除されたDNS検証レコード、CAAレコード、利用状況、メール検証の未承認などにより更新できない場合があります。RenewalStatus とEventBridgeの更新失敗イベントを監視します。

エクスポート可能な証明書

エクスポート可能なパブリック証明書は、2025年6月17日以降に、エクスポートを有効にして新規発行した証明書が対象です。発行後にエクスポート可否を変更できないため、既存の非エクスポート証明書は新しくリクエストします。

2026年7月時点の料金は、発行時と更新時にFQDNごとに7 USD、ワイルドカード名ごとに79 USDです。月10,000回までの証明書エクスポートAPI呼び出しは無料です。料金は変更される可能性があるため、導入時にAWS公式料金ページを確認します。

request
aws acm request-certificate \
  --domain-name example.com \
  --subject-alternative-names www.example.com \
  --validation-method DNS \
  --key-algorithm RSA_2048 \
  --options Export=ENABLED \
  --region ap-northeast-1

発行後はDNS検証を完了させます。ワイルドカードの *.example.com は example.com 自体を含まないため、両方を利用する場合はSANへそれぞれ指定します。

export
aws acm export-certificate \
  --certificate-arn <CERTIFICATE_ARN> \
  --passphrase fileb:///secure/path/passphrase.txt \
  --region ap-northeast-1 \
  > /secure/path/acm-export.json
  • パスフレーズファイルの末尾に改行を入れません
  • 証明書、証明書チェーン、暗号化された秘密鍵を分離して安全な場所へ配置します
  • 秘密鍵はrootなど必要な実行主体だけが読める権限にします
  • 設定変更後はApacheのconfigtestまたはnginx -tを通してから再読み込みます
  • 更新後も証明書ARNは変わりません。ファイルのシリアル番号と有効期限で更新を確認します

Apacheへの配置例とTLS設定は、Apache HTTP Server on RHEL 8 / 9 - TLSと証明書も参照します。

エクスポート証明書の自動再配置

AWS Workload Credentials Provider

AWS Workload Credentials Providerは、24時間ごととサービス起動時にACMから証明書を取得し、内容が変わった場合だけファイルを書き換えてrefresh_commandを実行します。EC2ではインスタンスプロファイルを基点に、証明書ごとのエクスポートロールを引き受ける構成にします。

config.toml example
[logging]
log_level = "info"
log_to_file = true

[capabilities.acm]
enabled = true

[[capabilities.acm.certificates]]
certificate_arn = "<CERTIFICATE_ARN>"
role_arn = "<ACM_EXPORT_ROLE_ARN>"
certificate_path = "/etc/pki/tls/certs/www.example.com.crt"
private_key_path = "/etc/pki/tls/private/www.example.com.key"
chain_path = "/etc/pki/tls/certs/www.example.com-chain.pem"
refresh_command = "/bin/systemctl reload httpd"

エクスポートロールには対象証明書のacm:ExportCertificateだけを許可し、インスタンスロールには対象ロールのsts:AssumeRoleだけを許可します。導入後はサービス状態、プロバイダーログ、ファイル権限、証明書の有効期限、Webサーバーの再読み込み結果を確認します。

EventBridge、Lambda、Systems Manager

独自配布が必要な場合は、ACMのACM Certificate AvailableイベントをEventBridgeで受け、更新時にLambdaから証明書をエクスポートし、Systems Managerで対象サーバーへ配布できます。

  1. EventBridgeでsource: aws.acm、detail-type: ACM Certificate Availableを受けます
  2. LambdaでイベントのActionがRENEWALであることと、対象の証明書ARNを確認します
  3. Secrets Managerからエクスポート用パスフレーズを取得します
  4. ExportCertificateで証明書、チェーン、暗号化された秘密鍵を取得します
  5. Systems Managerで一時ファイルへ配置し、所有者と権限を設定します
  6. apachectl configtestまたはnginx -t成功後に本番ファイルへ切り替えます
  7. サービスを再読み込みし、外部から証明書のシリアル番号と有効期限を確認します
  8. 失敗時は旧ファイルへ戻し、SNSなどへ通知します
⚠️ パスフレーズと秘密鍵をログへ残さない

パスフレーズをRun Commandのコマンド文字列へ直接埋め込むと、コマンド履歴や実行記録に残る可能性があります。Secrets Managerから対象ホストが必要時に取得する方式、Workload Credentials Provider、ACMEなど、秘密情報を平文で受け渡さない構成を優先します。

EventBridgeのイベント配信はベストエフォートです。イベント処理を冪等にし、CloudWatchアラームと定期確認を併用します。

ACM ACMEによるサーバー側自動更新

2026年7月から、ACMはACMEv2互換のマネージドACMEエンドポイントを提供しています。証明書は45日間有効で、Certbot、cert-manager、acme.shなどのACMEクライアントが発行と更新を行います。

2026年7月時点では、月内の最初の1,000 FQDNは発行または更新1回につき1 USD、最初の500ワイルドカード名は5 USDです。数量に応じた段階料金があり、ACME API呼び出しは月30,000回まで無料です。

  1. PKI管理者がACMでACMEエンドポイントを作成します
  2. 発行を許可するドメインと、完全一致・サブドメイン・ワイルドカードの範囲を指定します
  3. DNS検証用CNAMEを登録します
  4. External Account Binding（EAB）認証情報を作成し、ACMEクライアントを登録します
  5. Certbotなどで証明書を取得し、ApacheまたはNGINXへ設定します
  6. 更新後のdeploy hookで構文確認とサービス再読み込みを行います
  7. ACMEクライアントのタイマー、ログ、期限、CloudWatchメトリクスを監視します
Certbot command skeleton
certbot certonly \
  --server <ACM_ACME_ENDPOINT_URL> \
  --eab-kid <EAB_KEY_ID> \
  --eab-hmac-key <EAB_HMAC_KEY> \
  --email <ADMIN_EMAIL> \
  --agree-tos \
  -d www.example.com

ACME証明書の秘密鍵はACMEクライアント側で生成・保持され、ACMは秘密鍵を保持しません。ACMのマネージド更新やExportCertificateは使用せず、クライアント側の更新処理が証明書ライフサイクルを担当します。ACME証明書はALB、CloudFront、API GatewayなどのACM統合サービスへ関連付けられません。

期限監視と更新確認

監視対象確認内容
ACMDaysToExpiry、証明書ステータス、更新ステータス、DNS検証、利用先
EventBridge期限接近、証明書利用可能、更新操作要求、失効イベント
配布処理Lambda、Step Functions、Systems Manager、Workload Credentials Provider、ACMEクライアントの成否
サーバーファイル有効期限、シリアル番号、SAN、証明書と秘密鍵の対応、ファイル権限
外部エンドポイント実際に返される証明書、証明書チェーン、SNI、TLSハンドシェイク
server and endpoint verification
openssl x509 \
  -in /etc/pki/tls/certs/www.example.com.crt \
  -noout -subject -issuer -serial -dates -ext subjectAltName

openssl s_client \
  -connect www.example.com:443 \
  -servername www.example.com \
  -showcerts < /dev/null 2>/dev/null |
  openssl x509 -noout -subject -issuer -serial -dates

ACM上の証明書が更新済みでも、外部エンドポイントが古い証明書を返す場合があります。ACMと外部エンドポイントは別の監視対象として扱います。CloudWatchのAWS/CertificateManager名前空間では、DaysToExpiryが1日2回発行されます。
