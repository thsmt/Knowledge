Zabbix 運用手順

Zabbixの通知先変更、監視設定追加、Agent導入時の確認観点を整理します。画面手順は環境差分が出やすいため、作業前後の比較観点を中心にまとめます。

前提 RHEL9へZabbix Agent 2を導入する例です。root権限で実行する前提のため、コマンド例では sudo を省略しています。Zabbix 6.0系を利用する場合は、先に Zabbix公式ダウンロード に掲載されている公式リポジトリを追加します。

対象作業

作業確認観点
通知先変更ユーザー、メディア、送信先、通知テスト、作業前後差分を確認します。
監視設定追加Host、Template、Item、Trigger、Action、通知先の関連を確認します。
Agent導入Agentのバージョン、設定ファイル、Server/ServerActive、ListenPort、疎通を確認します。
監視削除削除対象、通知停止、依存テンプレート、復旧手順を確認します。

Zabbix Agent 2インストール

Zabbix 6.0のRHEL9向け公式リポジトリを追加します。dnf install だけでは通常Zabbix Agent 2を導入できないため、先に zabbix-release パッケージを導入します。

root shell
rpm -Uvh https://repo.zabbix.com/zabbix/6.0/rhel/9/x86_64/zabbix-release-6.0-5.el9.noarch.rpm

dnfのキャッシュを削除し、追加したリポジトリ情報を読み直します。

root shell
dnf clean all

Zabbix Agent 2をインストールします。

root shell
dnf install -y zabbix-agent2

Zabbix Agent 2設定

Agent設定ファイルを編集します。環境ごとにZabbix ServerのIPアドレスまたはFQDN、監視対象ホスト名を置き換えます。

値説明
<ZABBIX_SERVER_IP>Zabbix ServerまたはProxyのIPアドレス
<ZABBIX_SERVER_FQDN>Zabbix ServerまたはProxyのFQDN。IPで指定する場合は不要
<HOSTNAME>Zabbixに登録するHost名。管理画面側のHost名と一致させます。

設定ファイルを編集します。

root shell
vi /etc/zabbix/zabbix_agent2.conf

主な設定例です。パッシブチェックでは Server、アクティブチェックでは ServerActive を確認します。

conf
Server=<ZABBIX_SERVER_IP>
ServerActive=<ZABBIX_SERVER_IP>
Hostname=<HOSTNAME>

設定値を確認します。

root shell
grep -E '^(Server|ServerActive|Hostname)=' /etc/zabbix/zabbix_agent2.conf

Zabbix Agent 2確認

Zabbix Agent 2のバージョンを確認します。

root shell
zabbix_agent2 -V

Zabbix Agent 2を自動起動に設定し、起動します。

root shell
systemctl enable --now zabbix-agent2

サービス状態を確認します。

root shell
systemctl status zabbix-agent2

パッシブチェックを利用する場合は、Agentが10050/tcpで待ち受けていることを確認します。

root shell
ss -lntp | grep 10050
通信要件 パッシブチェックではZabbix ServerからAgentの10050/tcpへ到達できる必要があります。アクティブチェックではAgentからZabbix Serverへ接続します。Security Group、OS Firewall、ネットワークACL、経路をあわせて確認します。

通知先変更

通知先変更では、対象ユーザー、メディア種別、送信先、通知時間帯、有効/無効を作業前後で比較します。

  1. Zabbix管理画面にサインインします。
  2. 対象ユーザーまたはユーザーグループを開きます。
  3. メディア設定を確認し、作業前エビデンスを取得します。
  4. 通知先を追加または変更します。
  5. 更新後の設定と通知テスト結果を確認します。

監視設定追加

監視設定を追加する場合は、Host、Template、Item、Trigger、Actionを分けて確認します。作業後は最新データ、障害イベント、通知履歴を確認します。

エビデンス取得

タイミング取得対象
作業前対象Host、Template、Item、Trigger、Action、通知先
作業後変更後の設定、最新データ、通知テスト、イベント履歴
切り戻し時変更前設定への復旧、通知停止または復旧確認
