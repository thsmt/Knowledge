障害切り分けチェックリスト

ℹ️ 情報

このページは確認コマンドを中心に整理しています 復旧コマンドは原因と影響範囲を確認したあとに実行する前提で、各セクションの末尾に分けて記載する

⚠️ 注意

原因が見えていない段階で、サービス再起動、プロセス強制終了、OS再起動、ログ削除、SELinux無効化を先に行うと、 障害発生時の証跡が失われる場合がある

緊急復旧が必要な場合も、実行前後の時刻、コマンド、結果、判断理由を作業記録に残す

1. 初動で記録する情報

先に技術的なコマンドを実行するのではなく、発生状況を短く整理します 以降の調査では、この情報を基準に「対象が1台か複数台か」「OS内かAWS側か」「変更起因か」を切り分ける

項目記録内容
発生時刻タイムゾーンを含めて記録します。例: 2026-07-13 10:15 JST
対象ホスト名、IPアドレス、EC2 Instance ID、サービス名を記録する
OSRed Hat Enterprise Linux 9 / 10、Amazon Linux 2023などを確認する
事象エラーメッセージ、HTTPステータス、画面表示、ユーザー影響を記録する
影響範囲1ユーザーのみ、1台のみ、複数台、システム全体などを分ける
直前変更パッチ、設定変更、デプロイ、証明書更新、再起動、監視設定変更を確認する
再現性常時発生、断続的、特定時間帯のみ、特定経路のみなどを記録する

2. 60秒で行う基本状態確認

まずOSの稼働時間、時刻、負荷、メモリ、ディスク、失敗中のsystemdユニットを確認します ここでは復旧操作を行わず、全体像をつかむ

bash
date
hostnamectl
uptime
free -h
df -hT
df -ih
systemctl --failed
sudo journalctl -b -p err --since "-2 hours" --no-pager -o short-iso-precise
確認結果読み方次に見る項目
uptime のLoad Averageが高いCPUだけでなく、I/O待ちやプロセス滞留でも上がるCPU・Load Average、ディスク・inode・I/O
df -hT が100%に近いログ肥大化、バックアップ残骸、一時ファイルの滞留を疑うディスク・inode・I/O
systemctl --failed に対象サービスが出るsystemd管理のサービス起動失敗プロセス・systemd
journalctl -p err に同時刻のエラーがあるOS、カーネル、systemd、サービスの異常が時系列で確認できるログ確認
正常時・異常時の出力例

以下はサンプルです。実環境のホスト名、IPアドレス、Instance IDは記載していない

sample
# 正常例
$ uptime
 10:15:00 up 12 days,  2:10,  1 user,  load average: 0.12, 0.08, 0.04

$ systemctl --failed
  UNIT LOAD ACTIVE SUB DESCRIPTION
0 loaded units listed.

# 異常例
$ systemctl --failed
  UNIT          LOAD   ACTIVE SUB    DESCRIPTION
● httpd.service loaded failed failed The Apache HTTP Server

3. rsyslog / systemd-journal のログ確認

RHEL系とAmazon Linux 2023では、systemd-journalとrsyslogの使い方が環境により異なります まずjournalを時刻と重大度で確認し、必要に応じて /var/log/messages や /var/log/secure を確認する

bash
sudo journalctl -b -p err --since "-2 hours" --no-pager -o short-iso-precise

if [ -f /var/log/messages ]; then
  sudo grep -Ei 'error|fail|warn|oom|denied|read-only|no space' /var/log/messages | tail -n 100
fi

if [ -f /var/log/secure ]; then
  sudo grep -Ei 'failed|invalid|denied|authentication|pam|sudo|sshd' /var/log/secure | tail -n 100
fi
確認方法役割補足
journalctl -p errsystemd-journalから重大度で抽出するerr を指定すると、emerg、alert、crit、errも対象になる
grepテキストログを文字列で検索するseverityではなく文字列検索のため、検索語を変えながら確認する
/var/log/secureSSH、sudo、PAMなど認証系ログを確認する存在しない環境ではjournal側を確認する
ログ出力の読み方
sample
# 異常例: ディスク枯渇
2026-07-13T10:22:10+09:00 host.example systemd[1]: Failed to start The Apache HTTP Server.
2026-07-13T10:22:10+09:00 host.example httpd[1234]: No space left on device

# 異常例: 認証失敗
Jul 13 10:24:12 host sshd[2345]: Failed password for invalid user test from 203.0.113.10 port 55222 ssh2

サービス名、時刻、エラー文字列、直前変更を合わせて確認します No space left on device が出ている場合はサービス再起動より先にディスク容量を確認する

4. CPU・Load Average

Load Averageが高い場合でも、CPU使用率だけが原因とは限りません I/O待ち、プロセス滞留、短時間のスパイク、バックアップ処理、ログ圧縮などを分けて確認する

bash
uptime
top -b -n 1 | head -n 20
ps -eo pid,ppid,user,stat,pcpu,pmem,etime,cmd --sort=-pcpu | head -n 20
ps -eo pid,ppid,user,stat,pcpu,pmem,etime,cmd --sort=-pmem | head -n 20
見る値読み方次の確認
load averageCPU数に対して高いかを見ます。CPU数を超える状態が継続する場合は滞留を疑うtop、I/O待ち、プロセス状態
%Cpu(s) waI/O待ちです。CPUが空いていてもディスク待ちで遅くなる場合があるディスク・I/O
STAT が D割り込み不可スリープです。I/O待ちやNFS待ちなどを疑うディスク、ネットワーク、外部ストレージ

5. メモリ・Swap・OOM

メモリ不足では、プロセス停止、応答遅延、OOM Killerによる強制終了、swap多用による性能劣化が起こる

bash
free -h
swapon --show
ps -eo pid,ppid,user,stat,pcpu,pmem,rss,vsz,etime,cmd --sort=-rss | head -n 20
sudo journalctl -k --since "-24 hours" --no-pager | grep -Ei 'out of memory|oom|killed process|memory cgroup'
OOMのサンプル出力
sample
kernel: Out of memory: Killed process 12345 (java) total-vm:...
kernel: oom_reaper: reaped process 12345 (java)

OOMが出ている場合は、該当プロセス、発生時刻、直前のアクセス増加、メモリ上限設定、バッチ処理有無を確認します いきなりサービスを再起動すると、発生時のメモリ状態を追えなくなる

6. ディスク・inode・I/O

ディスク障害は容量不足だけではありません inode枯渇、読み取り専用マウント、I/Oエラー、ログ肥大、削除済みファイルをプロセスが掴んでいるケースを確認する

bash
df -hT
df -ih
sudo du -xhd1 /var 2>/dev/null | sort -h
findmnt -rn -o TARGET,SOURCE,FSTYPE,OPTIONS
findmnt -rn -o TARGET,SOURCE,FSTYPE,OPTIONS | grep -E '(^|[[:space:],])ro([,[:space:]]|$)'
sudo journalctl -k --no-pager | grep -Ei 'I/O error|read-only|xfs|ext4|nvme|blk_update_request'
症状読み方次に確認すること
容量100%ログ、バックアップ、一時ファイル、アプリ出力の肥大を疑うdu、logrotate、アプリログ
inode100%小さいファイルが大量に作成されているメールキュー、セッションファイル、キャッシュ
read-onlyファイルシステム保護やストレージ異常の可能性があるカーネルログ、EBS status checks
df と du が合わない削除済みファイルをプロセスが掴んでいる場合があるsudo lsof +L1
bash
sudo lsof +L1

7. プロセス・systemd

サービス単位の障害では、systemdの状態、直近ログ、設定ファイル構文、実行ユーザー、依存サービスを確認する

bash
systemctl status <SERVICE_NAME>
journalctl -u <SERVICE_NAME> -n 200 --no-pager
systemctl is-enabled <SERVICE_NAME>
systemctl cat <SERVICE_NAME>
systemctl show <SERVICE_NAME> -p User -p Group -p ExecStart -p Restart -p MemoryMax -p CPUQuota
手動実行では動くがsystemdでは失敗する場合

systemdでは環境変数、作業ディレクトリ、実行ユーザー、ulimit、PATHが対話シェルと異なることがあります systemctl cat と systemctl show でユニット定義を確認する

bash
systemctl show <SERVICE_NAME> -p Environment -p WorkingDirectory -p LimitNOFILE
sudo -u <SERVICE_USER> /path/to/command --check

8. ポート・TCP・TLS・HTTP

通信不可では、OS内部、Security Group、Network ACL、ルートテーブル、DNS、アプリケーション応答のどこで止まっているかを分ける

bash
ip addr
ip route
sudo ss -lntup
curl -vI --connect-timeout 5 --max-time 10 https://<FQDN>/
nc -vz -w 5 <HOST> <PORT>
確認結果読み方次の確認
localhostでは応答するアプリは起動しています。外部経路、SG、NACL、LB、DNSを確認するAWS側設定、LBヘルスチェック
ss にLISTENがない対象サービスがポート待受していないsystemd、アプリ設定、起動ログ
nc がtimeout経路、SG、NACL、FW、宛先待受のいずれかで止まっているルート、SG、NACL、宛先サーバー
curl がTLSで失敗証明書、SNI、プロトコル、時刻同期を疑う証明書期限・TLS、時刻同期

9. 名前解決・ルーティング

IPアドレスでは接続できるがFQDNでは失敗する場合、名前解決、検索ドメイン、キャッシュ、レコード変更の反映を確認する

bash
getent hosts <FQDN>
dig <FQDN>
dig +short <FQDN>
cat /etc/resolv.conf
ip route get <DESTINATION_IP>
名前解決の読み方

getent hosts はOSが実際に使う名前解決経路を確認します dig はDNS問い合わせ結果を確認します 両者が異なる場合は、/etc/nsswitch.conf、/etc/hosts、DNSキャッシュを確認する

10. Firewall・SELinux・ファイル権限

Permission deniedや接続拒否は、Linux権限、SELinux、firewalld、Security Groupを分けて確認します SELinuxを無効化する前に、拒否ログを確認する

bash
getenforce
sudo journalctl -t setroubleshoot --since "-2 hours" --no-pager
sudo ausearch -m avc,user_avc -ts recent 2>/dev/null
sudo firewall-cmd --state
sudo firewall-cmd --list-all
ls -lZ <PATH>
⚠️ 注意

setenforce 0 は原因切り分けとして使われることがありますが、恒久対応ではありません 実行する場合は、対象時刻、影響範囲、戻し手順、SELinux拒否ログを記録する

11. 時刻同期

Kerberos、TLS、署名付きAPI、ログ相関では時刻差が原因になります OS時刻とNTP同期状態を確認する

bash
timedatectl
chronyc tracking
chronyc sources -v

大きな時刻差を補正する chronyc makestep は、証跡とアプリ影響を確認してから実行する

12. 証明書期限・TLS

証明書更新後に古い証明書が返る場合、LB、Webサーバー、証明書チェーン、SNI、キャッシュを確認する

bash
echo | openssl s_client -servername <FQDN> -connect <FQDN>:443 2>/dev/null | openssl x509 -noout -subject -issuer -dates
curl -vI --connect-timeout 5 --max-time 10 https://<FQDN>/
症状確認観点
更新したのに古い証明書が出るALB/NLB、Webサーバー、CloudFront、SNI、接続先IPを確認する
一部端末だけ失敗する中間証明書、クライアントの信頼ストア、TLSバージョンを確認する

13. OSユーザーでログインできない

SSHログイン不可は、ネットワーク、sshd、OSユーザー、認証方式、鍵、PAM、SSSD、AD/LDAP、ホームディレクトリを分けて確認する

13.1 影響範囲を分ける

事象疑う範囲
全ユーザーがログインできないTCP/22、Security Group、sshd、PAM、ディスク枯渇、OS全体障害
特定ユーザーのみログインできないユーザー状態、パスワード期限、鍵権限、ホームディレクトリ、AllowUsers、SSSD
踏み台からのみ失敗接続元制限、Security Group、名前解決、踏み台側の鍵やSSHクライアント

13.2 OS側の確認

bash
sudo systemctl status sshd
sudo journalctl -u sshd --since "-2 hours" --no-pager
getent passwd <USER>
id <USER>
sudo passwd -S <USER>
sudo chage -l <USER>
sudo faillock --user <USER>
grep -E '^(AllowUsers|DenyUsers|AllowGroups|DenyGroups|Match)' /etc/ssh/sshd_config
sudo sshd -t

13.3 ホームディレクトリ・鍵・SELinux

bash
sudo ls -ld /home/<USER> /home/<USER>/.ssh
sudo ls -l /home/<USER>/.ssh/authorized_keys
sudo ls -Zd /home/<USER> /home/<USER>/.ssh /home/<USER>/.ssh/authorized_keys
df -hT /home
df -ih /home

13.4 AD / LDAP / SSSD利用時

bash
systemctl status sssd
sudo journalctl -u sssd --since "-2 hours" --no-pager
getent passwd <USER>
id <USER>
timedatectl
getent hosts <DOMAIN_CONTROLLER_FQDN>

14. EC2環境固有の確認

EC2ではOS内部だけでなく、Instance Status Checks、System Status Checks、EBS状態、Security Group、IAM Role、SSM Agentも確認する

14.1 インスタンスメタデータ

bash
TOKEN=$(curl -s -X PUT "http://169.254.169.254/latest/api/token" -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")
curl -s -H "X-aws-ec2-metadata-token: $TOKEN" http://169.254.169.254/latest/meta-data/instance-id
curl -s -H "X-aws-ec2-metadata-token: $TOKEN" http://169.254.169.254/latest/dynamic/instance-identity/document

14.2 AWS CLIでの確認

powershell
aws ec2 describe-instance-status `
  --region <REGION> `
  --instance-ids <INSTANCE_ID> `
  --include-all-instances

aws ec2 describe-volumes `
  --region <REGION> `
  --filters Name=attachment.instance-id,Values=<INSTANCE_ID> `
  --query "Volumes[].{VolumeId:VolumeId,State:State,Size:Size,Device:Attachments[0].Device,DeleteOnTermination:Attachments[0].DeleteOnTermination}"

aws ec2 describe-volume-status `
  --region <REGION> `
  --volume-ids <VOLUME_ID>
コマンド確認できること
describe-volumesEBSボリュームの設定、状態、サイズ、アタッチ先、終了時削除属性を確認する
describe-volume-statusEBSのステータスチェック、イベント、障害有無を確認する
describe-instance-statusEC2インスタンスのSystem Status / Instance Statusを確認する

14.3 SSM Agent

bash
systemctl status amazon-ssm-agent
sudo journalctl -u amazon-ssm-agent --since "-2 hours" --no-pager

SSM AgentがOfflineの場合は、IAM Role、VPC Endpoint、Outbound通信、DNS、時刻同期、Agentログを確認する

15. 運用でよくある質問

質問確認観点
Load Averageが高いがCPUは空いているI/O待ち、D状態プロセス、NFS/外部ストレージ待ちを確認する
df と du の容量が合わない削除済みファイルをプロセスが掴んでいないか lsof +L1 で確認する
inodeが100%になった小さいファイルの大量生成、メールキュー、セッション、キャッシュを確認する
Read-only filesystemになったカーネルログ、EBS status checks、ファイルシステムエラーを確認する
Too many open filesが出るLimitNOFILE、プロセスのFD数、アプリ設定を確認する
OOM Killerが動いた該当プロセス、メモリ推移、cgroup制限、アクセス増加を確認する
手動実行は成功するがsystemdでは失敗する環境変数、実行ユーザー、WorkingDirectory、PATH、ulimitを確認する
localhostでは動くが外部から接続できない待受アドレス、Security Group、NACL、LB、DNS、ルートを確認する
IPでは接続できるがFQDNでは失敗するDNS、/etc/hosts、検索ドメイン、キャッシュを確認する
証明書を更新したが古い証明書が返るALB、Webサーバー、CloudFront、SNI、接続先IPを確認する
cronやsystemd timerが動かないサービス状態、ユーザー、PATH、時刻、ログ、実行権限を確認する
sudoが突然失敗する/etc/sudoers 構文、グループ所属、LDAP/SSSD、PAMを確認する
SELinuxでPermission deniedになるAVCログ、ファイルコンテキスト、boolean、ポリシーを確認する
cloud-init / User Dataが失敗した/var/log/cloud-init.log、/var/log/cloud-init-output.log、IMDS、パッケージリポジトリを確認する
