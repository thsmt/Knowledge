障害切り分けチェックリスト

Red Hat Enterprise Linux 9 / 10、Amazon Linux 2023で障害が発生した際に、原因を体系的に切り分けるための確認手順を整理します。

📍 掲載方針

確認コマンドと復旧コマンドを混在させず、まずは参照系コマンドで事実を集めます。事実、仮説、実施内容、結果を分けて記録し、設定変更や復旧操作は影響範囲とロールバック方法を確認してから実行します。

⚠️ 注意

原因が判明する前にサービス再起動、プロセス強制終了、OS再起動、ログ削除などを実施すると、障害発生時の情報が失われる場合があります。

緊急復旧が必要な場合も、実行前後の時刻、コマンド、結果を作業記録に残します。

1. 初動で記録する情報

技術的なコマンドを実行する前に、発生状況を整理します。障害対応では「何が起きたか」だけでなく、「いつから、誰に、どこまで発生しているか」を分けて記録します。

項目確認内容
発生・検知時刻タイムゾーンを含めた正確な時刻。
対象サーバーホスト名、IPアドレス、EC2 Instance ID。
対象OSRed Hat Enterprise Linux 9 / 10、Amazon Linux 2023など。
事象エラー全文、画面表示、応答状況、ユーザー影響。
影響範囲1ユーザー、1サーバー、複数サーバー、システム全体。
最終正常時刻最後に正常動作を確認できた時刻。
直前変更パッチ、設定変更、証明書更新、デプロイ、再起動。
再現性常時、断続的、特定ユーザーのみ、特定時間帯のみ。

2. 基本状態の確認

OS、起動時刻、負荷、時刻同期の状態を最初に確認します。出力は障害発生時刻と照合できるように残します。

OSバージョンを確認します。

shell
cat /etc/os-release

起動時刻と稼働時間を確認します。直近で意図しない再起動が発生していないかを確認します。

shell
uptime
who -b

時刻同期の状態を確認します。ログ時刻のずれがあると原因調査が難しくなります。

shell
timedatectl
chronyc tracking

3. ログ確認

Red Hat Enterprise Linuxでは、一般的な構成で rsyslog が /var/log/messages や /var/log/secure にログを出力します。Amazon Linux 2023は標準状態で rsyslog が導入されていないため、まずログ管理方式を確認します。

ログ管理方式とJournalの状態を確認します。

shell
rpm -q rsyslog
systemctl is-active rsyslog
test -f /var/log/messages && echo "/var/log/messages exists" || echo "/var/log/messages does not exist"
journalctl --disk-usage
journalctl --list-boots

Journalから直近のエラーを確認します。emerg はSyslogの正式なプライオリティ名です。

shell
journalctl -p err..alert -n 200 --no-pager

RHEL系で /var/log/messages がある場合は、障害に関連しやすい文字列を検索します。

shell
grep -Eai 'emerg|alert|crit(ical)?|error|fail(ed|ure)?|fatal|panic|oom|out of memory|killed process|segfault|I/O error|read-only|corrupt' /var/log/messages

認証関連のログを確認します。SSHログイン不可、sudo失敗、アカウントロックの切り分けに使用します。

shell
journalctl -u sshd -n 200 --no-pager
grep -Eai 'failed|failure|invalid|denied|authentication|sudo|faillock' /var/log/secure

4. CPU・メモリ・ディスク

リソース逼迫は、アプリケーション障害、ログ出力停止、SSH接続不可など複数の症状につながります。現在値だけでなく、継続時間と急増の有無を確認します。

CPU、ロードアベレージ、プロセスの状態を確認します。

shell
uptime
top -b -n 1 | head -n 30
ps aux --sort=-%cpu | head -n 15

メモリ、swap、OOM Killerの発生有無を確認します。

shell
free -h
swapon --show
journalctl -k --no-pager | grep -Ei 'out of memory|oom|killed process'

ディスク使用率、inode、読み取り専用化の有無を確認します。

shell
df -hT
df -ih
mount | grep ' ro,'
journalctl -k --no-pager | grep -Ei 'I/O error|read-only|xfs|ext4|nvme|blk_update_request'

5. ネットワーク・名前解決

疎通不可は、OS内部、Security Group、Network ACL、ルートテーブル、DNS、アプリケーション待受のどこで止まっているかを分けて確認します。

IPアドレス、ルート、待受ポートを確認します。

shell
ip addr
ip route
ss -lntup

名前解決と外部疎通を確認します。

shell
getent hosts <FQDN>
dig <FQDN>
curl -I https://<FQDN>

対象ポートへTCP接続できるか確認します。

shell
nc -vz <HOST> <PORT>

6. サービス・プロセス

サービス単位の障害では、状態、直近ログ、設定ファイル構文、プロセスの有無を確認します。再起動は確認結果を残してから実施します。

サービス状態を確認します。

shell
systemctl status <SERVICE_NAME>
journalctl -u <SERVICE_NAME> -n 200 --no-pager

自動起動設定と失敗ユニットを確認します。

shell
systemctl is-enabled <SERVICE_NAME>
systemctl --failed

7. 症状別の切り分け

症状最初に見る場所次の確認
SSH接続不可sshd、認証ログ、Security Group。ポート22待受、鍵、ユーザー、Network ACL、踏み台経路。
Web応答不可ALB/NLB、Webサーバーサービス、待受ポート。HTTPステータス、アプリログ、Security Group、DNS。
ディスク逼迫df -hT、du、logrotate。急増ファイル、ローテーション失敗、バックアップ残骸。
CPU高騰top、ps、アプリログ。処理集中、無限ループ、外部接続待ち、バッチ処理。
メモリ不足free -h、OOMログ。プロセス別メモリ、swap、アプリ設定、再発周期。

8. EC2環境の確認

EC2ではOS内部だけでなく、AWS側のステータスチェック、EBS、Security Group、ルート、CloudWatchメトリクスも合わせて確認します。

インスタンスメタデータからInstance IDとリージョンを確認します。

shell
TOKEN=$(curl -s -X PUT "http://169.254.169.254/latest/api/token" -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")
curl -s -H "X-aws-ec2-metadata-token: $TOKEN" http://169.254.169.254/latest/meta-data/instance-id
curl -s -H "X-aws-ec2-metadata-token: $TOKEN" http://169.254.169.254/latest/dynamic/instance-identity/document

AWS CLIを利用できる場合は、ステータスチェックとEBS状態を確認します。

powershell
aws ec2 describe-instance-status --instance-ids <INSTANCE_ID> --include-all-instances
aws ec2 describe-volumes --filters Name=attachment.instance-id,Values=<INSTANCE_ID>

9. 復旧操作の扱い

復旧操作は、確認コマンドとは別に扱います。実行する場合は、影響範囲、ロールバック方法、作業時刻、実行者、実行結果を記録します。

⚠️ 注意

systemctl restart、kill -9、faillock --reset、setenforce 0、rm、xfs_repair、chronyc makestep、OS再起動は、原因調査に必要な状態を変える可能性があります。

実行前に、ログ、プロセス、メトリクス、設定ファイル、直前変更を確認します。

復旧操作実行前の確認
サービス再起動直近ログ、設定構文、依存サービス、影響範囲。
プロセス停止対象PID、親子関係、処理中データ、再起動方式。
OS再起動ステータスチェック、接続経路、起動設定、作業承認。
ログ削除取得済みエビデンス、圧縮退避、監査要件。
