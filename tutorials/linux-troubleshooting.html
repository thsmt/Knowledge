障害切り分けチェックリスト

Red Hat Enterprise Linux 9 / 10、Amazon Linux 2023、EC2上のLinuxサーバーで障害が発生したときに、 何を確認し、出力をどう読み、次にどのレイヤーを確認するかを整理します

ℹ️ 情報

このページは確認コマンドを中心に整理しています 復旧コマンドは原因と影響範囲を確認したあとに実行する前提で、各セクションの末尾または「復旧操作の扱い」に分けて記載します

⚠️ 注意

原因が見えていない段階で、サービス再起動、プロセス強制終了、OS再起動、ログ削除、SELinux無効化を先に行うと、 障害発生時の証跡が失われる場合があります

緊急復旧が必要な場合も、実行前後の時刻、コマンド、結果、判断理由を作業記録に残します

1. 初動で記録する情報

先に技術的なコマンドを実行するのではなく、発生状況を短く整理します 以降の調査では、この情報を基準に「対象が1台か複数台か」「OS内かAWS側か」「変更起因か」を切り分けます

項目記録内容
発生時刻タイムゾーンを含めて記録します。例: 2026-07-13 10:15 JST
対象ホスト名、IPアドレス、EC2 Instance ID、サービス名を記録します
OSRed Hat Enterprise Linux 9 / 10、Amazon Linux 2023などを確認します
事象エラーメッセージ、HTTPステータス、画面表示、ユーザー影響を記録します
影響範囲1ユーザーのみ、1台のみ、複数台、システム全体などを分けます
直前変更パッチ、設定変更、デプロイ、証明書更新、再起動、監視設定変更を確認します
再現性常時発生、断続的、特定時間帯のみ、特定経路のみなどを記録します

2. 60秒で行う基本状態確認

まずOSの稼働時間、時刻、負荷、メモリ、ディスク、失敗中のsystemdユニットを確認します ここでは復旧操作を行わず、全体像をつかみます

bash
date
hostnamectl
uptime
free -h
df -hT
df -ih
systemctl --failed
sudo journalctl -b -p err --since "-2 hours" --no-pager -o short-iso-precise
確認結果読み方次に見る項目
uptime のLoad Averageが高いCPUだけでなく、I/O待ちやプロセス滞留でも上がりますCPU・Load Average、ディスク・inode・I/O
df -hT が100%に近いログ肥大化、バックアップ残骸、一時ファイルの滞留を疑いますディスク・inode・I/O
systemctl --failed に対象サービスが出るsystemd管理のサービス起動失敗ですプロセス・systemd
journalctl -p err に同時刻のエラーがあるOS、カーネル、systemd、サービスの異常が時系列で確認できますログ確認
正常時・異常時の出力例

以下はサンプルです。実環境のホスト名、IPアドレス、Instance IDは記載していません

sample
# 正常例
$ uptime
 10:15:00 up 12 days,  2:10,  1 user,  load average: 0.12, 0.08, 0.04

$ systemctl --failed
  UNIT LOAD ACTIVE SUB DESCRIPTION
0 loaded units listed.

# 異常例
$ systemctl --failed
  UNIT          LOAD   ACTIVE SUB    DESCRIPTION
● httpd.service loaded failed failed The Apache HTTP Server

3. rsyslog / systemd-journal のログ確認

RHEL系とAmazon Linux 2023では、systemd-journalとrsyslogの使い方が環境により異なります まずjournalを時刻と重大度で確認し、必要に応じて /var/log/messages や /var/log/secure を確認します

bash
sudo journalctl -b -p err --since "-2 hours" --no-pager -o short-iso-precise

if [ -f /var/log/messages ]; then
  sudo grep -Ei 'error|fail|warn|oom|denied|read-only|no space' /var/log/messages | tail -n 100
fi

if [ -f /var/log/secure ]; then
  sudo grep -Ei 'failed|invalid|denied|authentication|pam|sudo|sshd' /var/log/secure | tail -n 100
fi
確認方法役割補足
journalctl -p errsystemd-journalから重大度で抽出しますerr を指定すると、emerg、alert、crit、errも対象になります
grepテキストログを文字列で検索しますseverityではなく文字列検索のため、検索語を変えながら確認します
/var/log/secureSSH、sudo、PAMなど認証系ログを確認します存在しない環境ではjournal側を確認します
ログ出力の読み方
sample
# 異常例: ディスク枯渇
2026-07-13T10:22:10+09:00 host.example systemd[1]: Failed to start The Apache HTTP Server.
2026-07-13T10:22:10+09:00 host.example httpd[1234]: No space left on device

# 異常例: 認証失敗
Jul 13 10:24:12 host sshd[2345]: Failed password for invalid user test from 203.0.113.10 port 55222 ssh2

サービス名、時刻、エラー文字列、直前変更を合わせて確認します No space left on device が出ている場合はサービス再起動より先にディスク容量を確認します

4. CPU・Load Average

Load Averageが高い場合でも、CPU使用率だけが原因とは限りません I/O待ち、プロセス滞留、短時間のスパイク、バックアップ処理、ログ圧縮などを分けて確認します

bash
uptime
top -b -n 1 | head -n 20
ps -eo pid,ppid,user,stat,pcpu,pmem,etime,cmd --sort=-pcpu | head -n 20
ps -eo pid,ppid,user,stat,pcpu,pmem,etime,cmd --sort=-pmem | head -n 20
見る値読み方次の確認
load averageCPU数に対して高いかを見ます。CPU数を超える状態が継続する場合は滞留を疑いますtop、I/O待ち、プロセス状態
%Cpu(s) waI/O待ちです。CPUが空いていてもディスク待ちで遅くなる場合がありますディスク・I/O
STAT が D割り込み不可スリープです。I/O待ちやNFS待ちなどを疑いますディスク、ネットワーク、外部ストレージ

5. メモリ・Swap・OOM

メモリ不足では、プロセス停止、応答遅延、OOM Killerによる強制終了、swap多用による性能劣化が起こります

bash
free -h
swapon --show
ps -eo pid,ppid,user,stat,pcpu,pmem,rss,vsz,etime,cmd --sort=-rss | head -n 20
sudo journalctl -k --since "-24 hours" --no-pager | grep -Ei 'out of memory|oom|killed process|memory cgroup'
OOMのサンプル出力
sample
kernel: Out of memory: Killed process 12345 (java) total-vm:...
kernel: oom_reaper: reaped process 12345 (java)

OOMが出ている場合は、該当プロセス、発生時刻、直前のアクセス増加、メモリ上限設定、バッチ処理有無を確認します いきなりサービスを再起動すると、発生時のメモリ状態を追えなくなります

6. ディスク・inode・I/O

ディスク障害は容量不足だけではありません inode枯渇、読み取り専用マウント、I/Oエラー、ログ肥大、削除済みファイルをプロセスが掴んでいるケースを確認します

bash
df -hT
df -ih
sudo du -xhd1 /var 2>/dev/null | sort -h
findmnt -rn -o TARGET,SOURCE,FSTYPE,OPTIONS
findmnt -rn -o TARGET,SOURCE,FSTYPE,OPTIONS | grep -E '(^|[[:space:],])ro([,[:space:]]|$)'
sudo journalctl -k --no-pager | grep -Ei 'I/O error|read-only|xfs|ext4|nvme|blk_update_request'
症状読み方次に確認すること
容量100%ログ、バックアップ、一時ファイル、アプリ出力の肥大を疑いますdu、logrotate、アプリログ
inode100%小さいファイルが大量に作成されていますメールキュー、セッションファイル、キャッシュ
read-onlyファイルシステム保護やストレージ異常の可能性がありますカーネルログ、EBS status checks
df と du が合わない削除済みファイルをプロセスが掴んでいる場合がありますsudo lsof +L1
bash
sudo lsof +L1

7. プロセス・systemd

サービス単位の障害では、systemdの状態、直近ログ、設定ファイル構文、実行ユーザー、依存サービスを確認します

bash
systemctl status <SERVICE_NAME>
journalctl -u <SERVICE_NAME> -n 200 --no-pager
systemctl is-enabled <SERVICE_NAME>
systemctl cat <SERVICE_NAME>
systemctl show <SERVICE_NAME> -p User -p Group -p ExecStart -p Restart -p MemoryMax -p CPUQuota
手動実行では動くがsystemdでは失敗する場合

systemdでは環境変数、作業ディレクトリ、実行ユーザー、ulimit、PATHが対話シェルと異なることがあります systemctl cat と systemctl show でユニット定義を確認します

bash
systemctl show <SERVICE_NAME> -p Environment -p WorkingDirectory -p LimitNOFILE
sudo -u <SERVICE_USER> /path/to/command --check

8. ポート・TCP・TLS・HTTP

通信不可では、OS内部、Security Group、Network ACL、ルートテーブル、DNS、アプリケーション応答のどこで止まっているかを分けます

bash
ip addr
ip route
sudo ss -lntup
curl -vI --connect-timeout 5 --max-time 10 https://<FQDN>/
nc -vz -w 5 <HOST> <PORT>
確認結果読み方次の確認
localhostでは応答するアプリは起動しています。外部経路、SG、NACL、LB、DNSを確認しますAWS側設定、LBヘルスチェック
ss にLISTENがない対象サービスがポート待受していませんsystemd、アプリ設定、起動ログ
nc がtimeout経路、SG、NACL、FW、宛先待受のいずれかで止まっていますルート、SG、NACL、宛先サーバー
curl がTLSで失敗証明書、SNI、プロトコル、時刻同期を疑います証明書期限・TLS、時刻同期

9. 名前解決・ルーティング

IPアドレスでは接続できるがFQDNでは失敗する場合、名前解決、検索ドメイン、キャッシュ、レコード変更の反映を確認します

bash
getent hosts <FQDN>
dig <FQDN>
dig +short <FQDN>
cat /etc/resolv.conf
ip route get <DESTINATION_IP>
名前解決の読み方

getent hosts はOSが実際に使う名前解決経路を確認します dig はDNS問い合わせ結果を確認します 両者が異なる場合は、/etc/nsswitch.conf、/etc/hosts、DNSキャッシュを確認します

10. Firewall・SELinux・ファイル権限

Permission deniedや接続拒否は、Linux権限、SELinux、firewalld、Security Groupを分けて確認します SELinuxを無効化する前に、拒否ログを確認します

bash
getenforce
sudo journalctl -t setroubleshoot --since "-2 hours" --no-pager
sudo ausearch -m avc,user_avc -ts recent 2>/dev/null
sudo firewall-cmd --state
sudo firewall-cmd --list-all
ls -lZ <PATH>
⚠️ 注意

setenforce 0 は原因切り分けとして使われることがありますが、恒久対応ではありません 実行する場合は、対象時刻、影響範囲、戻し手順、SELinux拒否ログを記録します

11. 時刻同期

Kerberos、TLS、署名付きAPI、ログ相関では時刻差が原因になります OS時刻とNTP同期状態を確認します

bash
timedatectl
chronyc tracking
chronyc sources -v

大きな時刻差を補正する chronyc makestep は、証跡とアプリ影響を確認してから実行します

12. 証明書期限・TLS

証明書更新後に古い証明書が返る場合、LB、Webサーバー、証明書チェーン、SNI、キャッシュを確認します

bash
echo | openssl s_client -servername <FQDN> -connect <FQDN>:443 2>/dev/null | openssl x509 -noout -subject -issuer -dates
curl -vI --connect-timeout 5 --max-time 10 https://<FQDN>/
症状確認観点
更新したのに古い証明書が出るALB/NLB、Webサーバー、CloudFront、SNI、接続先IPを確認します
一部端末だけ失敗する中間証明書、クライアントの信頼ストア、TLSバージョンを確認します

13. OSユーザーでログインできない

SSHログイン不可は、ネットワーク、sshd、OSユーザー、認証方式、鍵、PAM、SSSD、AD/LDAP、ホームディレクトリを分けて確認します

13.1 影響範囲を分ける

事象疑う範囲
全ユーザーがログインできないTCP/22、Security Group、sshd、PAM、ディスク枯渇、OS全体障害
特定ユーザーのみログインできないユーザー状態、パスワード期限、鍵権限、ホームディレクトリ、AllowUsers、SSSD
踏み台からのみ失敗接続元制限、Security Group、名前解決、踏み台側の鍵やSSHクライアント

13.2 OS側の確認

bash
sudo systemctl status sshd
sudo journalctl -u sshd --since "-2 hours" --no-pager
getent passwd <USER>
id <USER>
sudo passwd -S <USER>
sudo chage -l <USER>
sudo faillock --user <USER>
grep -E '^(AllowUsers|DenyUsers|AllowGroups|DenyGroups|Match)' /etc/ssh/sshd_config
sudo sshd -t

13.3 ホームディレクトリ・鍵・SELinux

bash
sudo ls -ld /home/<USER> /home/<USER>/.ssh
sudo ls -l /home/<USER>/.ssh/authorized_keys
sudo ls -Zd /home/<USER> /home/<USER>/.ssh /home/<USER>/.ssh/authorized_keys
df -hT /home

14. EC2環境固有の確認

EC2ではOS内部だけでなく、Instance Status Checks、System Status Checks、EBS状態、Security Group、IAM Role、SSM Agentも確認します

14.1 インスタンスメタデータ

bash
TOKEN=$(curl -s -X PUT "http://169.254.169.254/latest/api/token" -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")
curl -s -H "X-aws-ec2-metadata-token: $TOKEN" http://169.254.169.254/latest/meta-data/instance-id
curl -s -H "X-aws-ec2-metadata-token: $TOKEN" http://169.254.169.254/latest/dynamic/instance-identity/document

14.2 AWS CLIでの確認

powershell

16. 障害に備えた事前準備

障害発生後に情報が不足すると、原因特定よりも証跡集めに時間を使います 事前に以下を整備しておくと、初動が安定します

準備目的
CloudWatch Agent / rsyslog転送OSログ、アプリログをサーバー外へ退避します
journal永続化再起動後もjournalを確認できるようにします
CPU / Memory / Disk / inode監視リソース枯渇を早期検知します
OOM / systemd failed監視プロセス強制終了やユニット失敗を検知します
証明書期限監視TLS期限切れを事前に検知します
Chrony同期監視時刻差による認証・TLS・ログ相関不備を避けます
logrotate設計ログ肥大化によるディスク枯渇を防ぎます
平常時ベースラインCPU、メモリ、ポート、プロセス、ログ量の通常値を把握します
sysstat / sar / PCP過去時点のリソース状態を確認できるようにします

17. 初動証跡収集スクリプト

障害発生直後に最低限の情報を同じ形式で残すためのサンプルです 実行前に、保存先と取得対象に機密情報が含まれないか確認します

bash
OUT_DIR="/tmp/troubleshoot-$(hostname)-$(date +%Y%m%d-%H%M%S)"
mkdir -p "$OUT_DIR"

{
  date
  hostnamectl
  uptime
  free -h
  df -hT
  df -ih
  ip addr
  ip route
  systemctl --failed
} > "$OUT_DIR/basic.txt" 2>&1

sudo journalctl -b -p err --since "-2 hours" --no-pager -o short-iso-precise > "$OUT_DIR/journal-errors.txt" 2>&1
ps -eo pid,ppid,user,stat,pcpu,pmem,etime,cmd --sort=-pcpu | head -n 50 > "$OUT_DIR/ps-cpu.txt"
ps -eo pid,ppid,user,stat,pcpu,pmem,etime,cmd --sort=-pmem | head -n 50 > "$OUT_DIR/ps-mem.txt"

tar -czf "$OUT_DIR.tar.gz" -C "$(dirname "$OUT_DIR")" "$(basename "$OUT_DIR")"
echo "$OUT_DIR.tar.gz"
⚠️ 注意

収集したファイルには、ユーザー名、プロセス引数、内部IP、ホスト名などが含まれる場合があります 外部共有前にマスクします

18. 復旧操作の扱い

復旧操作は、確認コマンドとは分けて扱います 実行する場合は、影響範囲、実行理由、戻し方、作業前後の確認結果を残します

操作実行前に確認すること
サービス再起動直近ログ、設定構文、依存サービス、利用者影響、再起動後の確認手順
プロセス停止対象PID、親子関係、処理中データ、再起動方式、証跡取得済みか
OS再起動ステータスチェック、ディスク状態、アプリ影響、起動順序、メンテナンス承認
ログ削除取得済みエビデンス、圧縮退避、削除対象の妥当性、調査要件
SELinux一時変更AVCログ、対象パス、恒久対策、戻し手順、監査観点
faillock解除対象ユーザー、失敗回数、攻撃兆候、本人確認、解除後の監視

19. OSごとの差分メモ

項目Red Hat Enterprise Linux 9 / 10Amazon Linux 2023
ログsystemd-journalとrsyslog構成を確認します環境によりjournal中心の構成になるため、/var/log/messages の有無を確認します
SELinux有効化されている環境が多いため、getenforce で確認します初期状態やAMI差分があるため、前提にせず getenforce で確認します
パッケージsubscription-managerや社内リポジトリ設定を確認しますAmazon Linuxリリースバージョンと dnf のreleaseverを確認します
SSM Agent標準導入でない場合がありますAMIにより導入済みの場合がありますが、サービス状態を確認します
